# 模型结构演化与取舍

## 功能概述

本项目的模型结构选择，不是从“哪个网络更新”出发，而是从“当前误差到底是数据、损失函数、测量口径还是结构失配”出发。

这份文档用于收敛长期稳定的结构判断：

- 为什么早期纯 LSTM 基线最终不够
- 为什么路线逐步收敛到线性动态先验 + 可训练非线性主体
- 什么情况下该换模型，什么情况下不该先动模型
- 做 FRIKAN 系列消融时，如何保证语义等价

具体模型项目、单次调参结果和一次性实验记录，仍分别留在对应的专题文档中。

## 结构演化主线

### 早期基线：纯时序网络先验证数据链路

早期 LSTM 基线的主要价值，是先确认以下问题已经打通：

- 时域数据可以稳定组织和加载
- 模型可以直接吃下时序数据并完成训练
- 频率响应可以从模型输出重新测量出来

因此，LSTM 在项目早期是必要的工程基线，但它不是长期主路线的终点。

### 中期判断：不要把准线性问题当纯黑盒时序回归

随着 NOTEBOOK 中的多轮对比逐步稳定，问题被重新定义为“幅度依赖频率响应”的准线性系统，而不是普通时间序列拟合。

这个判断带来两个后果：

- 结构设计要优先保留线性动态骨架
- 非线性部分应更多承担幅度相关修正，而不是从零学出整个系统

### 后续收敛：线性前端 + 可训练主体

在这个问题定义下，长期更可复用的路线变成：

- 用 IIR、状态空间或等效 RNN 表达线性动态前端
- 用 KAN、MLP 或其他可训练主体表达非线性修正
- 用统一频响链路验收，而不是只看训练期 loss

仓库里的 FRIKAN、FRIMLP、FRIKAND，都是这一路线下的不同主体替换或消融，而不是彼此无关的独立模型家族。

## 什么时候不要先换模型

以下情况优先不要把问题归因于模型结构：

- 训练与评估只有在某台机器或某份旧 cache 上才能复现
- 序列开头异常明显，且问题集中在冷启动段
- 时域表现正常但频响异常，且尚未核对统一测量口径
- 不同模型差距很小，但数据覆盖范围或 loss 设计仍明显不足

这类问题更常见的根因是：

- 特征缓存键不完整
- 起始段窗口不完整
- 频响测量方法、缩放或反缩放有误
- 数据范围或损失函数没有覆盖目标现象

也正因此，NOTEBOOK 中曾明确提出“神经网络训练的三要素是数据、模型、损失函数；当模型差距不大时，先从数据和损失函数入手”。

数据集覆盖、稳态片段和低震级样本平衡的长期规则，详见 [dataset_design.md](dataset_design.md)。

## 各类结构的长期定位

### 纯 LSTM / GRU

适合：

- 作为数据链路和训练链路的早期基线
- 验证模型是否能捕捉基本的时间依赖
- 作为与带线性先验模型对比的参照组

不适合直接作为长期主路线的情况：

- 系统存在明显谐振峰和窄带动态
- 需要稳定表达中心频率漂移和峰值变化
- 已经确认问题更像准线性动态系统，而不是任意黑盒映射

GRU 可以作为同类替代尝试，但它改变的是循环单元细节，不改变“是否缺少线性动态先验”这一根问题。

### 纯 FIR 前端

纯 FIR 在当前问题上不应作为默认优先项。

原因不是 FIR 不能用，而是对高 Q 谐振峰和反馈型动态而言，低阶 FIR 通常既不经济，也不够可解释。若问题核心就是窄带共振和频率漂移，应优先考虑 IIR 或其等效形式。

### IIR 或等效 RNN 前端

当目标系统存在以下特征时，应优先考虑这一路线：

- 明显谐振峰
- 中心频率漂移
- 幅度相关的窄带动态
- 希望前端与系统辨识、频响分析结果直接对应

在项目里，引入 IIR 的核心意义不是“必须用 IIR 这个名字”，而是引入一个可解释的线性单元。如果工程实现上更适合写成无激活 RNN，只要时间步严格对齐，本质上仍属于同一类前端。

### FRIKAN 系列

FRIKAN 系列的长期价值在于：

- 把线性动态先验显式保留在前端
- 让非线性主体只承担剩余修正
- 为结构消融提供稳定对照面

因此，FRIKAN 更接近“带强先验的混合模型”，而不是单纯换一种神经网络层。

### FRIMLP / FRIKAND 等消融变体

这类模型只在一个前提下有意义：

- 保持 FRIKAN 的前端语义不变，只替换主体

如果连前端、fast_model、系统初始化路径也一起改掉，那么结果就不再是“主体消融”，而变成了新的混合实验。

因此，做这类消融时应优先检查：

- 是否仍走同一套 FRI 前端构建路径
- `prepare_systems()` 是否仍被调用
- `H_UNITS` 是否仍对应前端通道数
- fast_model 是否仍与基线一致

详细修复约束见 [frimlp_ablation.md](frimlp_ablation.md)。

### CNNKAN / 卷积变体

CNNKAN 这类结构更适合测试局部时域模式提取是否有帮助，或者作为主体替换、局部先验注入的工程尝试。

但它不能天然替代线性谐振先验。如果问题根因在于“模型没有表达好窄带共振和反馈动态”，仅把主体换成卷积或卷积 + KAN，通常不是第一优先级。

它更适合回答的问题是：

- 局部卷积特征是否有助于当前数据形态
- 在前端语义不变或问题已简化后，主体替换是否能提升效果

### Transformer 等替代结构

Transformer 或带注意力的时序模型，只在以下前提下才值得优先推进：

- 已确认长距离上下文是当前主要瓶颈
- 线性前端、数据覆盖和损失函数设计已基本稳定
- 评估链路已经证明当前失败不是频响测量或缩放问题

否则，它们更适合作为探索性分支，而不是主线补救手段。

## 结构调整的优先顺序

当当前模型效果不理想时，长期建议按下面顺序排查：

1. 先确认数据、缓存、预处理和评估口径没有问题。
2. 再确认损失函数是否真的对目标频响现象敏感。
3. 然后判断现有结构是否缺少线性动态先验。
4. 只有在前三步都较清晰时，才继续替换主体或尝试新架构。

这能避免把本来属于数据或评估链路的问题，误判成“模型不够新”。

## 做结构消融时的约束

- 只改用户要求改的结构变量，其他条件保持不变。
- 做局部替换时，优先守住语义等价，不要顺手改训练路径或系统初始化。
- 训练前先确认 `model_info.json` 中的前端输出形状与设计语义一致。
- 对比结论以统一评估链路和 `metrics.json` 为准，不以训练期 loss 代替。

## 推荐工作流

1. 用简单基线先打通数据与训练链路。
2. 用仿真系统判断问题是否真的需要线性动态先验。
3. 若答案是需要，则优先引入 IIR 或等效 RNN 前端。
4. 以 FRIKAN 作为带前端先验的主对照结构，再做主体消融。
5. 只有当主体能力成为瓶颈时，才继续尝试 CNNKAN、GRU、Transformer 等替代结构。
