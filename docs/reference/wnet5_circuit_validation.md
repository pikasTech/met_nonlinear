# WNET5 电路分层验证

## 功能概述

WNET5 电路验证的长期目标，不是直接比较“整网最后输出像不像”，而是把理论模型、TF/NumPy 推理、SPICE 仿真和实测频响拆成可定位的层级验证链路。

这份文档收敛以下长期稳定规则：

- WNET5 分层验证的推荐顺序
- SVF 层的通道映射、反相和 Q 值处理原则
- 自测试补偿与实验频响对比的口径
- Project 权重加载、E96 量化误差仿真和报告产物约束
- 哪些偏差应先归因于电路实现，哪些才值得回到模型本身

## 当前代码与入口

与 WNET5 电路分层验证直接相关的入口如下：

- `src/visualization/wnet5_circuit_validator.py`：理论频响、实验对比、自测试补偿与统计
- `src/inference/wavenet5_spice_backend.py`：WaveNet5 专用 SPICE 推理后端
- `ex_projects/wnet5-circuit-validation/layer1`
- `ex_projects/wnet5-circuit-validation/layer2`
- `ex_projects/wnet5-circuit-validation/layer3`

常用命令入口为：

- `python cli.py ep "ex_projects/wnet5-circuit-validation/layer1"`
- `python cli.py ep "ex_projects/wnet5-circuit-validation/layer2"`
- `python cli.py ep "ex_projects/wnet5-circuit-validation/layer3"`

EP 的目录、产物路径和运行方式以 [ep.md](ep.md) 为准；本文件只收敛长期验证规则。

与自动化回归入口相关的 pytest / `cli.py --test` 约定，详见 [testing.md](testing.md)；本文件不重复维护测试框架用法。

## 自动化回归与人工验收的分工

WNET5 分层验证相关代码修改后，长期上应把“自动化回归通过”和“分层图验收通过”视为两件事：

- 自动化回归负责尽早发现模块导入、SPICE 导出接口和基础对象装配是否被改坏。
- 分层验证文档负责规定图、权重口径、通道映射和误差统计是否真的满足当前任务语义。

也就是说，pytest 通过不能直接替代 `layer1/layer2/layer3` 的频响与实测对照；但如果连基础自动化回归都不过，就不应继续解释分层图结果。

## 推荐验证顺序

WNET5 电路验证应固定按以下顺序推进：

1. 先确认理论 SVF / Dense 响应的公式与参数口径。
2. 再确认 TF / NumPy 逐层推理输出是否一致。
3. 再确认 SPICE 分层输出是否与 TF / NumPy 对齐。
4. 最后才用实测频响验证硬件实现。

不要在前面三步未对齐时，直接拿实测曲线给整网下结论。

## SVF 层是第一验收门槛

当前 WNET5 路线里，SVF 层必须先被单独验证通过，原因是：

- 它承担线性前端的大部分物理语义
- 一旦 SVF 层的通道顺序、相位或 Q 值口径不对，后面的 Dense 与激活层误差都会被污染
- `wnet5_circuit_validator.py` 已经为 SVF 单独提供了理论频响、实测对比和补偿入口

因此，长期规则是：

- 先看 SVF 层
- 再看 Dense 层
- 最后才看整链输出

## 通道映射必须显式定义

### 不要默认不同链路的通道顺序天然一致

WNET5 的 TF、NumPy、SPICE 和实测数据，长期上都不能假设它们天然使用同一个输出顺序。

特别是 SVF 层，至少要明确以下三件事：

- 当前链路输出的是哪组滤波器类型顺序
- 实测文件的通道顺序是什么
- 对比时是否已经做了显式重排

只要通道语义没明确，任何误差图都不能直接解释为“模型不对”或“电路不对”。

### 实测映射要写在验证层，而不是靠人脑记忆

`wnet5_circuit_validator.py` 已经把实测文件命名、层号和通道号解析做成显式逻辑。长期原则是：

- 映射规则应当写进验证层代码或配置
- 不要靠临时脚本或人工记忆去对齐通道
- 同一批图的映射规则必须可复现

## 相位与反相要先校准再比幅值

在 WNET5 / SVF 路线下，部分通道出现反相并不意味着系统错误，更常见的是相位约定尚未统一。

长期规则是：

- 先确认哪些通道在电路实现上天然反相
- 再决定是在 SPICE 输出端、后处理层还是理论对照层做统一反转
- 相位约定未统一前，不要直接比较时域误差或频响误差百分比

对于 SVF 层，这一步通常先于 Dense 层分析。

## Q 值与带通增益的长期判断

### Q 不为 1 时不能偷用简化假设

当前已经收敛的长期结论是：

- Q 值不为 1 时，SVF 带通通道的增益口径必须显式包含 Q
- 不能把带通分量默认当成已归一化通道
- 若理论曲线只在 Q=1 时正常，而在 Q≠1 时失真，先查公式和参数映射

`wnet5_circuit_validator.py` 中 `_get_svf_magnitude_response()` 已经按显式公式处理 `HP / BP / LP` 幅值响应，这一口径应视为当前权威实现。

### 先修理论口径，再讨论器件偏差

如果 Q 值相关误差还停留在理论模型或参数换算层，就不应过早归因于运放、容差或仿真器。

## 自测试补偿的使用边界

### 自测试补偿用于剥离测量链路误差

当 `svf_error_simulation.compensation.enabled` 打开时，当前逻辑会用自测试频响对实测曲线做相除补偿。

长期目的不是“把图修好看”，而是：

- 先剥离测量链路自身响应
- 再看被测 SVF / Dense 结构本身的偏差

### 补偿前提

自测试补偿只有在以下前提满足时才有解释价值：

- 自测试文件与实验文件覆盖相近频段
- 插值后的频点口径一致
- 自测试对象和实验测量链路属于同一硬件路径

如果频率范围都对不上，补偿结果只能作为粗略参考。

## 实验文件扫描与命名约定

当前验证器已支持两类实验文件命名：

- `output_{时间戳}_SVF-W_DENSE{层号}_{通道号}_震级X.xlsx`
- `output_{时间戳}_SVF-W_DENSE{层号}-{通道号}_震级X.xlsx`

长期要求是：

- 层号和通道号必须能从文件名稳定解析
- 自测试文件应与实验文件区分命名
- 路径中若含环境变量，应通过统一展开逻辑处理，而不是手工改脚本

## 权重来源与路径口径

### 分层验证权重必须来自 project 产物

WNET5 分层验证里，Dense 权重的长期权威来源应是训练项目导出的权重 JSON，而不是手工编辑在 EP 配置里的常量。

长期规则是：

- `wnet5-circuit-validation` 配置应指向具体 project，再由验证器从 project 产物读取 Dense 权重。
- 读取来源优先是项目内已导出的 JSON 权重，而不是为了取权重再额外依赖 TensorFlow runtime。
- 手工填写单层权重只可视为早期历史痕迹，不应继续作为可复用方案。

这样做的目的不是“统一写法”，而是保证分层验证与真实训练项目保持同一组权重口径，避免配置漂移。

### 仓库内路径用相对路径，仓库外数据用环境变量

长期路径规则如下：

- 仓库内的 project、EP 模板、报告和图片路径应使用仓库相对路径。
- 仓库外的实验数据根目录应通过环境变量展开，例如 `MET_DATA_BASE`。
- 不要把某台机器的绝对盘符路径固化到 `config.json` 或报告模板里。

这能保证 WNET5 验证任务在不同机器间迁移时，真正需要改的只有环境变量，而不是一批散落的配置路径。

## E96 量化误差仿真

### 长期工作流

WNET5 Dense 层的 E96 量化误差仿真，应固定沿用下面这条链路：

1. 从 project 权重导出原始权重矩阵。
2. 将原始权重映射为浮点电阻值。
3. 将浮点电阻映射到目标标准系列，例如 `E96`。
4. 从量化后的电阻反推带量化误差的权重矩阵。
5. 用“原始权重仿真”和“带量化误差权重仿真”走同一条分层验证与绘图路径。

长期上不应跳过中间态，也不应只比较电阻表而不回算权重矩阵；真正要进入仿真的是“回算后的量化权重”，不是电阻表本身。

### 方向性与误差定义

量化分析里，电阻值虽然是正实数，但展示口径必须保留权重的方向性语义。

长期规则是：

- 电阻热力图中的符号应跟随原始权重方向，便于直接观察正负映射关系。
- 量化相对误差应以“原始权重矩阵 vs 回算后的量化权重矩阵”为准，而不是以“浮点电阻 vs E96 电阻”为准。
- 若图上宣称量化误差极小，必须能用 dump 出来的中间 JSON 反证该结论，而不是凭目测或对 E96 表的直觉判断。

### 热力图与频响图产物约束

E96 量化分析的长期产物要求如下：

- 量化对比图应在同一张大图中包含 5 个子图：原始权重、浮点电阻、E96 电阻、量化权重、量化相对误差。
- 每个子图都应与 Dense 权重矩阵同形，例如 6x6；如果出现缺格、维度不一致或部分数值缺失，应直接视为实现错误。
- 原始权重与量化权重必须共用同一配色和刻度；浮点电阻与 E96 电阻也应共用同一配色和刻度。
- 每个网格中都应直接标注数值，避免只看颜色无法判断量化偏差。

频响对照图则应复用现有 `frequency_response_comparison_merged` 风格，而不是另起一套临时绘图逻辑。长期关注点是“量化前后仿真链路是否只改了权重”，而不是图的样式差异。

## SVF 实测误差建模

### 优先使用拟合传递函数，而不是“实测增益 + 理论相位”拼接

SVF 误差仿真的长期口径，应以“拟合得到的实测传递函数参数 + 现有理论仿真代码”作为权威路径。

不再推荐的旧思路是：

- 幅值直接用实测曲线
- 相位继续借用理论公式

这类拼接口径虽然能快速出图，但会把“测量偏差”和“模型结构偏差”混在一起，不适合作为长期基线。

### 拟合顺序与验收顺序

当前更稳妥的长期顺序是：

1. 先拟合 SVF 的实测幅频响应。
2. 初值优先使用理论参数，而不是完全随机初值。
3. 先验证“拟合后的 SVF-only 曲线”与实测是否一致。
4. 再把同一套拟合参数送入 SVF + Dense 链路做最终对比。

当拟合质量不足时，优先先把幅值拟合对齐；相位不是第一优先级。也就是说，长期默认是“先拟合幅值，再决定是否需要额外引入相位建模复杂度”。

## 通道数不匹配时的处理

WNET5 分层验证中，SVF 展开通道数与 Dense 权重输入通道数并不总是天然一致。

长期规则是：

- 若 SVF 通道数多于权重输入通道数，应显式裁剪。
- 若 SVF 通道数少于权重输入通道数，应按已有通道循环扩展。
- 两种情况都必须给出 warning，而不是静默处理。
- 只有在无法确定映射语义时，才应终止任务。

这里的核心目标是保证任务可继续运行并暴露口径风险，而不是因为通道数不一致就直接把整条验证链路阻断。

## 报告生成约束

WNET5 分层验证若生成 Markdown 报告，长期上应遵守以下约束：

- 只生成一个总报告，并在报告中插入该次运行生成的全部关键图片。
- 图片插入必须使用相对路径 Markdown 链接，并在生成后校验所有 PNG 都已被正确引用。
- 报告文字说明图片用途、横纵轴和曲线来源即可，不要把图中的数值再手工抄写一遍。
- 若任务包含器件反推，应把理论值、标称值和拟合反推值的对比表一并纳入总报告，而不是散落到额外临时文件。

## 误差统计口径

当前验证器的长期口径是：

- 先把理论或仿真结果插值到实测频点
- 在同一频点上计算误差比值 `sim / exp`
- 再输出 `mean_error_ratio`、`mean_abs_error_percent`、`within_5pct`、`within_10pct` 等统计

长期解释原则是：

- 统计结果只能在“通道映射、相位约定、自测试补偿”都已处理后才有意义
- 如果这些前置条件没满足，统计值只能用于发现异常，不能用于下结论

## Dense 层与高通偏差的优先归因

器件非理想、收敛报错和偏置来源的更具体规则，详见 [spice_device_bias_practices.md](spice_device_bias_practices.md)。

当 WNET5 分层验证进入 Dense 或高通补偿环节后，长期优先排查项如下：

- 大阻值高通或分压网络造成的负载效应
- 偏置路径把增益整体拉向 0
- 前一层误差在下一层被继续放大

也就是说，如果高通或 Dense 结果明显往 0 收敛，第一反应应当是检查电路实现与静态工作点，而不是直接回头怀疑训练损失。

## 哪些问题优先归因于电路而不是模型

以下情况应优先从电路实现排查：

- SVF 层在理论、TF、NumPy 已一致，但 SPICE 或实测不一致
- Q≠1 时带通增益异常
- 被动高通或大阻值分压导致响应整体塌缩
- 更换器件模型后偏置或低频误差剧烈变化

以下情况才值得优先回查模型侧：

- TF 与 NumPy 的逐层输出本身就不一致
- scaler、输入数据集或层级切分口径发生变化
- 同一层在纯软件路径下已出现不可解释的语义漂移

## 推荐工作流

1. 用 `layer1` 先跑 SVF 理论 / SPICE / 实测对照。
2. 明确通道映射和相位约定后，再进入 `layer2` / `layer3`。
3. Dense 或高通偏差优先查负载、偏置和静态工作点。
4. 只有前端层级都对齐后，才拿整链结果讨论补偿能力。

若本轮修改同时涉及 `wnet5_circuit_validator.py`、WaveNet5 SPICE 导出或相关 EP 调度，建议先跑 [testing.md](testing.md) 中的定向 WNET5 / SPICE 回归，再继续执行上述分层工作流。
