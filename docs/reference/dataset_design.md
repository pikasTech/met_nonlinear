# 数据集设计与覆盖边界

## 功能概述

本项目的数据集设计目标，不是简单堆更多样本，而是让训练数据真正覆盖“幅度依赖频率响应”这一目标现象。

这份文档用于收敛长期稳定的数据规则：

- 训练数据应覆盖哪些维度
- 为什么稳态片段比整段原始波形更重要
- 单频、多频、变幅、变频数据各自适合回答什么问题
- 哪些现象更像是数据覆盖问题，而不是模型结构问题
- 当前数据集支持哪些结论，不支持哪些外推

具体项目的数据路径、波形导出和临时实验记录，仍分别留在命令文档和 NOTEBOOK 中。

## 数据集演化主线

从 NOTEBOOK 的长期演化看，数据集方向经历了几个稳定收敛：

- 从早期少量样本和单一条件，逐步转向多频、多震级覆盖
- 从直接处理原始长时域波形，逐步转向以稳态片段为主的数据组织
- 从弱标签数据，逐步转向显式保留频率标签、震级标签和可追溯参数
- 从“只看能不能训练起来”，转向“数据是否足以支撑频响结论”

因此，当前项目更看重的是覆盖性、标签完整性和可复现性，而不只是样本总数。

## 训练数据需要覆盖的核心维度

### 频率覆盖

数据必须覆盖目标工作频段内的主要频点，尤其是：

- 谐振峰附近
- 低频端和高频端边界
- 指标计算或物理解释最敏感的关键频点

如果只在少量频点表现良好，不足以说明模型已经学会整个频带的补偿规律。

### 震级覆盖

幅度依赖频率响应的核心就在于不同震级下频响会变化，因此震级覆盖不能只是“顺带多采几个值”，而应视为与频率同等重要的主维度。

长期规则是：

- 数据中必须包含多个震级层级，而不是只围绕单一工作点
- 低震级和高震级都要有足够代表样本
- 如果目标是跨震级补偿，就不能只用单一震级训练再期待自然泛化

### 频率与震级的联合覆盖

仅有“多频”或“多震级”都不够。更重要的是两者的联合覆盖，也就是：

- 不同震级下都能观察同一频率区域的变化
- 不同频率下都能观察震级变化对系统的影响

这也是变幅变频扫频数据比单一条件数据更有长期价值的原因。

## 稳态片段优先于整段原始波形

对当前问题，训练数据优先以稳态片段为主，而不是整段原始扫频或拼接波形直接喂给模型。

原因包括：

- 频响分析本身主要依赖稳态响应
- 起始瞬态、切换瞬态和截断冲击容易污染目标现象
- 不同脚本对瞬态的处理方式不一致时，很容易引入伪差异

因此，长期建议是：

- 数据裁剪、淡入淡出、稳态窗口选择要有统一口径
- 如果训练目标是稳态频响，不要让明显的起止冲击主导 loss
- 只有在明确研究瞬态行为时，才单独保留整段瞬态数据作为专门数据集

时域预处理和稳态频响测量的统一实现口径，详见 [timeseries_frequency_analysis.md](timeseries_frequency_analysis.md)。

## 单频数据与多频数据的定位

### 单频数据

单频数据更适合：

- 验证模型是否能学会基本的动态映射
- 做 IIR / RNN 前端等价性检查
- 快速判断结构是否可训练

但单频训练更容易给出“能拟合”的假象，不足以支撑多频带补偿结论。

### 多频数据

多频数据更适合：

- 验证模型在整个目标频带上的一致性
- 评估谐振峰附近和频带边界的泛化能力
- 支撑统一频响指标的最终结论

长期经验是：单频数据往往更容易训练，多频数据更接近真实目标，也更容易暴露结构、数据和损失函数的真实短板。

## 低震级样本不能长期缺位

NOTEBOOK 后续多轮结果都指向同一个长期问题：低震级样本不足或权重失衡时，模型很容易在高震级上看起来不错，但在小信号区域明显发飘。

因此，数据设计时应注意：

- 不要让高震级样本在数量或幅度上完全主导训练
- 对低震级段要保证足够采样密度
- 如果低震级误差长期偏大，先检查样本覆盖和权重，再判断是否是模型问题

这类问题常常还会和损失函数设计耦合在一起；相关原则见 [loss_design.md](loss_design.md)。

## 噪声、滤波与截断处理

数据质量不只是“噪声越少越好”，而是处理方式不能改变要学习的目标现象。

长期规则如下：

- 对纯净仿真数据，不要默认沿用面向噪声数据的滤波设置
- 如果滤波会改变 offset、相位或稳态幅值，应先确认它不是在破坏训练目标
- 去掉滤波后如果出现尾部截断冲击，应优先通过分组、裁剪或淡出解决，而不是直接忽略

简言之，滤波、裁剪和淡出都属于数据定义的一部分，而不是随手可改的可视化小修饰。

## 真实数据与合成数据的分工

数据设计层面，合成数据更适合做机制隔离，实验台真实数据更适合支撑当前工程结论。

但“当前数据到底能支持哪些结论、不能外推到哪些真实场景”的统一边界，不再在本文件重复展开，详见 [validation_boundaries.md](validation_boundaries.md)。

## 数据集复现与缓存

数据集定义一旦变化，旧 cache 就可能让训练和评估得到貌似同配置、实则不同口径的结果。

长期规则是：

- 特征缓存键必须覆盖全部会影响结果的数据生成条件
- 改了频点、时间步、裁剪长度、滤波、降采样或标签组织方式后，必须重新核对缓存口径
- 发现 `-t` 与 `-e` 只能在某份历史 cache 上复现时，优先排查数据定义和 cache，而不是先怀疑随机性

训练链路中的 cache 与起始段约束，详见 [training.md](training.md)。

## 当前数据集能支持什么结论

当前数据集主要回答“数据是否覆盖了目标现象”，而不是单独定义整个项目的对外验证边界。

关于当前实验结果能支持什么结论、不能直接外推出什么结论，详见 [validation_boundaries.md](validation_boundaries.md)。

## 推荐工作流

1. 先用单频或简化数据集打通训练链路与前端实现。
2. 再用多频多震级稳态数据验证结构是否真的覆盖目标现象。
3. 用合成数据隔离结构或 loss 的作用，再回到真实扫频数据验证。
4. 每次改动数据定义后都重新核对 cache、标签和稳态窗口。
5. 最终结论仍以统一评估链路和 `metrics.json` 为准。
