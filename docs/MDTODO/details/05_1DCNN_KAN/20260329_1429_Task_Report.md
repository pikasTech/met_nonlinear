## 问题 3：评估指标不一致 [严重]

### 现象
CNNKAN 和 FRIKAN 的 `linearity_by_frequency.json` 数据结构不同：

**FRIKAN：**
```json
{
  "gains_origin": [...],   // 增益 dB 值
  "gains_comped": [...],
  "r_squared_origin": 0.684,
  "r_squared_comped": 0.319,
  "improvement": -0.365
}
```

**CNNKAN：**
```json
{
  "output_rms_origin": [...],  // RMS 值
  "output_rms_comped": [...],
  "r_squared_origin": 0.965,
  "r_squared_comped": 0.873,
  "improvement": -0.092
}
```

### 分析
1. **指标类型不同**：FRIKAN 用增益（dB），CNNKAN 用 RMS
2. **R² 值含义不同**：无法直接对比两个模型的线性度
3. **improvement 含义不同**：由于基线不同，改善百分比不能直接比较

### 影响
- 无法客观评估 CNNKAN 相对于 FRIKAN 的性能优劣
- 可能影响模型选择和超参数调优决策

### 建议
统一评估指标，使用相同的 R² 计算方法和物理量单位。

## 问题 4：CNNKAN 性能问题 [需进一步调查]

### 现象
CNNKAN 所有频率的 `improvement` 均为负值，表示补偿后线性度变差：

| 频率(Hz) | R²_origin | R²_comped | improvement |
|----------|-----------|-----------|-------------|
| 10.0 | 0.965 | 0.873 | **-0.092** |
| 13.0 | 0.962 | 0.425 | **-0.537** |
| 16.0 | 0.972 | 0.511 | **-0.461** |
| ... | ... | ... | ... |

### 分析
- 原始系统已有较高线性度（R² > 0.93）
- 补偿后 R² 反而下降
- **可能原因**：
  1. 模型架构不适合该任务
  2. 训练不充分（只有 10 epochs）
  3. CNN 的卷积核大小（3）和通道数（8）不适合当前数据特征
  4. 补偿方向错误

### 注意
FRIKAN 的 `improvement` 也有大量负值，但 FRIKAN 是用 dB 计算的，CNNKAN 用 RMS，不能直接对比。

## 问题 5：fast_model 架构设计 [中等]

### 代码分析
`frikan_models.py` 第 891-899 行：
```python
self.fast_cnn = tf.keras.layers.Conv1D(
    filters=cnn_filters,
    kernel_size=cnn_kernel_size,
    ...
    name='fast_cnn_filter'
)
self.fast_iir = self.fast_cnn  # 别名
```

### 问题
1. `fast_cnn` 和 `cnn` 是**独立的层实例**，不共享权重
2. `fast_iir` 只是 `fast_cnn` 的别名，这是为了兼容 BaseModel 的接口
3. 这导致 `fast_model` 期望 8 通道输入，而 `model` 期望 1 通道输入

### CNNKAN 的解决方案
重写 `fit()`, `evaluate()`, `predict()`, `save_weights()` 方法，直接使用 `self.model` 而非 `self.fast_model`。

### 风险
这种设计使得 `use_fast_model=True` 时实际不使用快速路径，可能导致：
- 训练/推理效率低于预期
- 与 BaseModel 的设计意图不符
