# R24.4 任务报告：Table 5 `Constr. / Odd / No positive basis` 指标爆炸诊断

## 任务目标

诊断 Table 5 中 `Constr. / Odd / No positive basis` 这一组为什么会出现明显的物理指标恶化，顺着对应项目的 `config.json` 生效链路深入到代码实现，确认是否存在配置未生效或实现 bug。

## 结论

本轮结论是：**没有发现 `config.json -> model_subcfg -> FRIKAN -> DenseKAN` 这条链路的实现 bug。**

`Table 5` 里这一行当前来自：

- `docs/paper/config.json`
- `structure_ablation -> label = "No positive (stress)"`
- 对应项目：`projects/01_LR_STUDY/FRIKANh8u6l6_e1k_lr7e4_nopositive`

该项目的 `config.json` 中：

```json
"model_subcfg": {
  "only_positive": false,
  "use_even": false,
  "use_symmetry": true
}
```

这组配置在当前实现中的真实语义是：

- **保留 odd symmetry**；
- **去掉 positivity projection**；
- **不是 fully unconstrained，也不是配置失效。**

因此，当前指标爆炸更合理的解释是：**去掉 positivity 约束后，KAN 正半轴允许学习到带负值/反向弯折的奇对称映射，频响补偿函数明显失稳，最终把频漂、灵敏度漂移和线性度一起拉坏。**

## 代码链路核查结果

我沿着配置生效路径逐段核查了以下位置：

1. `projects/01_LR_STUDY/FRIKANh8u6l6_e1k_lr7e4_nopositive/config.json`
   - `model_subcfg.only_positive = false`
   - `model_subcfg.use_even = false`
   - `model_subcfg.use_symmetry = true`
2. `src/config.py`
   - `Config.load_from_json()` 会把 `model_subcfg` 原样载入配置对象。
3. `src/core/model_engine.py`
   - `build_model()` 在创建 `FRIKAN` 时，直接传入 `model_subcfg=self.config.model_subcfg`。
4. `src/models/frikan_models.py`
   - `FRIKAN.__init__()` 会读取 `only_positive/use_even/use_symmetry`；
   - 同一组布尔值同时传给输出层 `self.kan` 和全部 `self.kan_inner_layers`。
5. `src/tfkan/layers/base.py`
   - `LayerKAN.calc_spline_output_symmetry()` 中，**只有** `only_positive=true` 时才会执行 `self.spline_kernel.assign(tf.abs(self.spline_kernel))`；
   - `only_positive=false` 时不会做这一步，负 spline 系数会被保留；
   - `use_even=false` 时会乘回 `sign(inputs)`，因此最终仍是 odd symmetry。
6. `src/tfkan/layers/dense.py`
   - `DenseKAN.compute_spline_output()` 在上述 spline 分支输出之后，再叠加 `basis_activation`；当前项目 `FIX_SCALE_FACTOR=true`，因此这里没有额外的可学习缩放因子把布尔语义覆盖掉。

结论：**配置没有在中途丢失，也没有被默认值偷偷覆盖。**

## 直接诊断证据

### 1. 当前 Table 5 这一行确实来自 `lr7e4_nopositive`

`docs/paper/config.json` 明确把 `No positive (stress)` 指向：

- `projects/01_LR_STUDY/FRIKANh8u6l6_e1k_lr7e4_nopositive/data/metrics.json`

而当前 `metrics.json` 中的核心数值为：

- `Freq Drift (Hz) = 58.03`
- `Sens Drift (%) = 50.87`
- `Linearity (%) = 38.586`

因此 Table 5 里的爆炸值不是 paper pipeline 拼错了项目，而是确实来自这一条 `nopositive` 训练产物。

### 2. 现有 `nopositive` 学习率扫过多个点，但都没有接近基线

我补查了 `projects/01_LR_STUDY/FRIKANh8u6l6_e1k_lr*_nopositive/data/metrics.json`，当前已存在的几个 `nopositive` 变体在新指标口径下依然整体很差：

- `lr1e4_nopositive`: `Freq Drift = 304304.52`
- `lr14e5_nopositive`: `Freq Drift = 629794.24`
- `lr2e4_nopositive`: `Freq Drift = 277798.57`
- `lr3e4_nopositive`: `Freq Drift = 352662.51`
- `lr4e4_nopositive`: `Freq Drift = 658252.94`
- `lr5e4_nopositive`: `Freq Drift = 217.58`
- `lr7e4_nopositive`: `Freq Drift = 58.03`

其中当前 Table 5 选用的 `lr7e4_nopositive` 反而是这批现有 `nopositive` 里相对最不差的一条，不是“刚好挑中了最坏点”。

### 3. 在真实训练路径中，`only_positive=false` 的分支确实会放行负映射

我在 `tf26` 环境里分别构建了：

- baseline：`projects/01_LR_STUDY/FRIKANh8u6l6_e1k_lr7e4`
- odd-only：`projects/01_LR_STUDY/FRIKANh8u6l6_e1k_lr7e4_nopositive`

并加载已训练权重，直接抽样 KAN 层正半轴映射。

关键结果：

- baseline：输出层 probe 的 `neg_ratio = 0.0`
- `nopositive`：输出层 probe 的 `neg_ratio = 0.7743`

也就是说，在 `nopositive` 已训练权重上，**正半轴采样点里约 77% 的输出贡献已经掉到负值区域**；而 baseline 的对应 probe 没有出现这种现象。

这与代码逻辑完全一致：

- baseline 因为 `only_positive=true`，forward 时会把 `spline_kernel` 投影到非负；
- `nopositive` 因为 `only_positive=false`，负系数被保留，最终允许学出 sign-changing 的 odd 映射。

所以这不是“配置没生效”，而是“配置生效后把模型确实带进了不稳定函数形态”。

## 最终判断

本轮没有发现需要立即修复的实现 bug；当前更准确的判断是：

1. `Table 5` 里的 `Odd / No positive basis` 不是 paper pipeline 指错项目；
2. 也不是 `config.json` 的布尔开关没有生效；
3. 它反映的是 **“去掉 positivity、保留 odd symmetry”** 这条消融本身就很不稳定；
4. 因此当前这一行的爆炸值是**模型行为问题**，不是**配置接线 bug**。

唯一需要额外注意的是写作语义：这一行更精确的描述应是：

- `w/o positivity constraint, keep odd symmetry`

而不是：

- `fully unconstrained`
- `remove all constraints`

否则后续正文解释会把结论说得比代码实际实现更强。
