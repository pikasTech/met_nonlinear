# R13.2 任务报告

- 时间：2026-04-23
- 任务：优化 WIENER-KAN + LUT 的板端推理速度，重点检查 LUT 路径是否存在算法级低效，并给出可验证结果。
- 结论：原 LUT 路径确实存在明显的算法/数据布局低效。完成优化后，`project_default` 的板端实测速度从此前已发布的 `1861.946 Points/s` 提升到本次手动复测的 `6048.701 Points/s`，约为 `3.25x`。

## 1. 问题定位

这次不是简单的编译优化问题，核心瓶颈主要有两个：

1. 旧实现把 `LUT map -> 边界判断 -> index 计算 -> 对称符号处理` 放在“每条 edge lookup”上重复执行。
   - 对 `FRIKANh8u6l6` 这类多层小宽度网络来说，单点前向虽然层数不大，但 edge 数很多；旧实现会把同一个输入在同一层上的 LUT map 重复做多次。
   - 这意味着慢的不只是 LUT 读表本身，而是“同一输入重复 map + 重复分支”的开销被放大了。

2. 旧 LUT 存储布局对板端访问模式不友好。
   - 原布局是 `frikan_layer_lut[layer][input][output][lut_point]`。
   - 板端真正需要的访问模式是：固定 `layer + input + lut_point`，一次性拿到这个 input 对所有 outputs 的贡献。
   - 旧布局下，不同 output 的同一 `lut_point` 彼此跨着整段 LUT，等价于大跨度从 Flash 取数；这对板端非常不友好。

所以用户的直觉是对的：`LUT MAP` 本身不该慢成之前那样，真正的问题是“重复 map + 错误的数据访问方向”。

## 2. 实施修改

本次改动集中在 `src/core/board_inference/models/frikan.py`，没有改训练主流程，也没有动正式论文 `main.tex`。

### 2.1 LUT 数据布局改成 point-major 输出复用布局

把 LUT 存储从：

- 旧：`[layer][input][output][lut_point]`

改成：

- 新：`[layer][input][lut_point][output]`

这样在板端对某个 `input` 完成一次 LUT map 之后，就能顺序读出该 `lut_point` 下所有 outputs 的贡献，避免跨整段 LUT 跳着取数。

### 2.2 把“每条边重复 map”改成“每个输入只 map 一次”

为每一层生成专门化的 LUT 累加函数：

- `frikan_lut_accumulate_fast_layer_X(outputs, input_index, raw_value)`

它的行为是：

1. 只对当前 `input` 做一次 `abs/map/bounds/index`；
2. 按新的 LUT 布局顺序遍历 outputs；
3. 直接把该 input 对整层 outputs 的贡献一次性累加进输出缓冲区。

也就是说，旧路径是“每个 output 各自做一遍 map”，新路径是“map 一次，复用给该 input 的所有 outputs”。

### 2.3 前向代码生成改成 input-major

LUT 路径的前向生成由原来的 output-major 展开，改成：

1. 先初始化一层的局部 `outputs` 缓冲；
2. 再按 `input_index` 循环；
3. 每个 input 调一次对应层的 `frikan_lut_accumulate_fast_layer_X(...)`。

这个结构和 LUT 的真实访问模式一致，也更利于后续继续做单 profile benchmark 或进一步手工优化。

### 2.4 保持 exact 非 LUT 路径不变

本次只改 LUT 路径；exact lookup 路径保持原逻辑，避免把这次速度结论和 exact 路径耦合在一起。

### 5.2 手动有效签收路径

为避免被批量烧录链路干扰，本次采用了更稳的单 profile 实测路径：

1. `keil-cli detect` 确认 probe 与 `STM32F405RGTx` 链路正常；
2. 用 `pyocd` backend 手动烧录默认 profile；
3. 启动串口监控；
4. 抓取板端输出中的完整 `FRIKAN_BENCHMARK_VALIDATION`；
5. 选取“最终 reset/go 之后的最后一个完整 block”作为有效结果。

有效证据已拷贝到仓库内：

- `ex_projects/inference/qemu-c-inference/frikan_h8u6l6_e1k_lr7e4/data/manual_pyocd_project_default_benchmark_20260423.json`

## 7. 结论与后续建议

### 7.1 本任务结论

R13.2 的核心目标已经完成：

- 证明了原 LUT 路径确实存在算法级低效；
- 完成了 LUT 路径的结构性优化，而不是只靠编译器参数碰运气；

### 7.2 当前遗留问题

当前剩下的问题不是 LUT 算法本身，而是自动 `ep keil-bench` 的多 profile 连续烧录链路稳定性：

- `Keil UV4` 路径会出现 `RDDI-DAP Error`
- `pyOCD` 批量 profile 路径会出现偶发 `SWD/JTAG communication failed`

这属于工具链/探头/连续烧录流程层面的抖动，与本次 LUT 算法优化结论分离。
