## 根因调查

本轮排查发现，问题并不在于 `model_compute_analysis` 的主公式本身没有更新，而在于**默认配置层仍把旧值重新覆写回去了**：

1. `src/analysis/model_compute_analysis.py` 的默认权重已经是 `add:multiply:MAP = 1:4:23`；
2. 但 `src/config.py` 里的 `compute_cost_model` 默认值仍然是旧的 `1:1:6`；
3. 项目执行 `cli.py -m`、`--metrics` 等默认流程时，会从 `Config` 读取该字段，再传给 compute analysis；
4. 结果就是：虽然长期文档和部分代码已经写成 `1:4:23`，但很多历史重新导出的 `compute_analysis.json` 仍被旧默认值覆写，实际落盘的 `platform_cost_model.weights` 还是 `1:1:6`。

这也解释了此前 draft 中一个很迷惑的现象：有些产物的 `basis` 文案已经写着 `1:4:23`，但真正用于汇总的权重字段仍停留在旧值，导致表格与图里的 `Compute Cost` 仍是陈旧结果。

## 关键结果

### 1. 主横评静态复杂度已更新为真正的 `1:4:23` 结果

重刷后，主横评的 `Compute Cost` 为：

- Wiener-KAN: `5808`
- TCN: `10656`
- 1DCNN: `4924`
- LSTM: `9216`
- LSTMTransformer: `9676`
- RNN: `3456`
- GRU: `7184`

与旧 draft 里残留的 `1710 / 3904 / 1880 / ...` 相比，这组数值才是真正通过默认 CLI 流程落盘的最新口径。

### 2. Wiener-KAN 相对 TCN 的复杂度优势仍然成立，但幅度回归到更真实的水平

在修正默认值覆写后：

- Wiener-KAN 相对 TCN 的 `Compute Cost` 降低 `45.50%`
- Wiener-KAN 相对 TCN 的 `KEIL-SPEED` 提速 `49.74%`

也就是说，Wiener-KAN 仍然保有“更低静态复杂度 + 更快板端时延”的结论，但幅度不再是旧稿中的 `56.20%`，而是与实际落盘口径一致的 `45.50%`。

### 3. 旧稿里的趋势错位大部分已经被解释

修正后可以看到：

- TCN 重新成为主部署子集里**最高静态 `Compute Cost`**、同时也是**最慢板端时延**的模型；
- LSTMTransformer 的静态复杂度重新回到与 LSTM 接近的区间，不再像旧稿那样看起来明显“算得偏低”。

因此，R10.1 里观察到的那类“TCN / LSTMTransformer 明显趋势不一致”现象，至少在当前 draft 使用的数据链上，已经可以主要归因为**旧默认值覆写导致的陈旧复杂度产物**，而不是必须继续追查的主公式错误。

### 4. 标定结果本身没有变，变化的是它终于真正进入了默认流程

本轮并没有推翻 R10 已得到的标定结论：

- 采用默认权重：`1:4:23`
- 对数域 `RMSE`：`0.2334 -> 0.1168`
- 最大相对误差：`42.16% -> 20.22%`

真正修复的是：这套标定结果过去只存在于文档、分析脚本和部分产物中，但没有被完整贯彻到默认 CLI 刷新链；现在它已经真正进入默认流程。
