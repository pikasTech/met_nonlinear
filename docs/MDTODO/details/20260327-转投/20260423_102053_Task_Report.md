## 关键结果与结论

### 1. 多优化档结果已经成为正式产物，而不是临时 benchmark

R13.1 的关键价值，不只是“跑了三档编译器开关”，而是把这三档结果纳入了可重复引用的正式数据链：

- 上游部署 summary 负责保存各 profile 的 build / run / validation / size 结果；
- `metrics.json` 负责保存 published profile 的默认板端字段，并把 profile 列表作为补充信息保留；
- draft `3.4` 直接从正式产物引用，不再依赖临时控制台输出。

### 2. 失败 profile 不再打断整轮 benchmark

旧式工作流的风险是：只要某一档构建失败，整轮 on-board benchmark 就被打断，最后既没有对照，也没有明确失败记录。现在的正式行为已经变成：

1. 单个 profile 可失败；
2. 失败结果会被写成显式状态，例如 `build_failed`；
3. 其余 profile 仍继续执行并保留成功结果；
4. 论文可以把“失败本身”作为工程边界的一部分直接写出来。

对于本任务而言，这一点非常重要，因为 `-O0` 的“Flash overflow”本身就是值得保留的结果。

### 3. `On board 推理性能评估` 的写法已经更工程化

完成 R13.1 之后，`3.4` 节不再只是“谁快谁慢”的部署子表，而是同时回答四个问题：

1. 各模型在 QEMU / Keil 下与 TensorFlow 参考路径是否一致；
2. 各模型的板端吞吐和资源占用分别是多少；
3. Wiener-KAN 在不同编译优化档下是否还能继续提速；
4. 提速的边界是否被 Flash / RAM / build failure 所限制。

这使 `3.4` 节真正具备了 `On board 推理性能评估` 的含义，而不是简单的“跑一个真机 benchmark 截图”。
