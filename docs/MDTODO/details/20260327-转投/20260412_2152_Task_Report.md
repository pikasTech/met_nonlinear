# R3.4 任务报告：转投 MN 是否需要大幅补强消融实验与横评

## 任务概述

本任务结合以下材料，判断当前主稿若转投 `Microsystems & Nanoengineering`（MN），是否有必要把消融实验与横向对比补强到更接近 `AI_Based_Sensor` 那类 AI+工程论文的强度：

- 当前主稿：`docs/paper/main.tex`
- 转投主旨：`docs/IDEA.md`
- 拒稿意见分析：`docs/MDTODO/details/20260327-转投/20260327_1803_Task_Report.md`
- MN 对标差距分析：`docs/MDTODO/details/20260327-转投/20260412_1754_Task_Report.md`
- AI+领域结构分析：`docs/MDTODO/details/20260327-转投/20260412_1829_Task_Report.md`
- MN 领域匹配再判断：`docs/MDTODO/details/20260327-转投/20260412_1838_Task_Report.md`
- MN 已发表 AI+论文借鉴：`docs/MDTODO/details/20260327-转投/20260412_1927_Task_Report.md`

本报告要回答的问题是：

**如果目标是让当前稿件达到更接近 `AI_Based_Sensor` 的实验成熟度，那么转投 MN 时，是否有必要显著补强横评与消融实验？如果有，应该补强到什么方向、什么程度？**

---

## 一、结论先行

本次 R3.4 的核心结论可以压缩为四句话：

1. **是，有必要显著补强横评和消融实验。** 如果目标是对齐 `AI_Based_Sensor` 这类 MN 中较成熟的 AI+工程论文，那么当前主稿的实验矩阵明显偏弱，尤其缺少结构化横评、机制性消融和更严格的验证协议。
2. **主题修改同样是必须的，而且和实验补强是绑定关系。** 不能只在旧版 FRIKAN 叙事上机械加实验；必须同时把论文主轴改成 MN 更接受的 `physical sensor / calibration / linearization / dynamic range extension` 叙事。
3. **因此，正确方向不是“只改主题”或“只补实验”，而是两条线同时推进。** 主题负责决定论文被当作什么类型的工作理解，实验负责给这个新主旨提供足够硬的证据。
4. **补实验要向 `AI_Based_Sensor` 的实验设计强度靠拢，但不是盲目堆模型名，而是建立一套完整、层次清楚的横评 + 消融 + 严格验证框架。**

换句话说：

- **主题重写是必要条件。**
- **更强的横评和消融是必要支撑。**
- **两者缺一不可。**

---

## 二、为什么现在必须改口：当前稿件实验强度确实不够

R1 的拒稿意见已经说明，上一稿被追问的重点不是单一图表问题，而是整套实验设计不够充分：

- 没有原始 KAN 对比；
- 没有物理约束组合消融；
- 没有频响注入消融；
- 没有完整损失函数消融；
- 对比模型数量有限；
- 缺少更接近 SOTA 的横向比较；
- 数据集划分与鲁棒性说明偏弱；
- 计算成本对比不完整。

而当前 `main.tex` 仍然保留了会继续触发这些问题的写法：

- 标题和摘要仍以 FRIKAN/PIKAN/FRIRNN 为主角：`docs/paper/main.tex:124`、`docs/paper/main.tex:138`
- 贡献条目仍以模型和损失函数创新为中心：`docs/paper/main.tex:203`
- 结果解释仍把收益直接归因于 FRIRNN、PIKAN、AFMAE：`docs/paper/main.tex:1063`、`docs/paper/main.tex:1109`
- 数据划分仍是随机切分子序列：`docs/paper/main.tex:1026`

所以，如果要继续冲 MN，而且希望稿件实验部分达到更接近 `AI_Based_Sensor` 的成熟度，那么继续维持当前这种实验规模是不够的，这一点应当明确接受，而不是回避。

---

## 三、为什么“主题修改”和“实验补强”必须一起做

### 3.1 只补实验、不改主题，会越补越像旧版方法稿

如果继续把论文写成：

- FRIKAN 是新模型；
- PIKAN 是关键创新；
- FRIRNN 是关键创新；
- AFMAE 是关键创新；

那补实验之后，稿件虽然会更完整，但整体仍然会像一篇 method-heavy 的 AI 论文。这样做的后果是：

- 审稿人会继续按“新模型论文”的标准要求更多对比；
- 论文重心会进一步远离 MN 更偏好的器件/系统主线；
- 补实验的边际收益会不断下降，因为你会被拖入一个没有上限的“模型自证”路径。

### 3.2 只改主题、不补实验，会让新主旨缺乏支撑

反过来，如果只把标题、摘要、引言改成：

- calibration；
- linearization；
- dynamic range extension；
- sensor-side deployment；

但实验部分依然只有当前这种较薄的矩阵，那么新主旨也站不住。因为新的叙事会自然要求：

- 你相对已有方法到底强在哪；
- 你的关键设计是否真的必要；
- 你的校准框架是否可靠、稳定、可复现；
- 你的低复杂度结论是否由完整证据支持。

因此，**主题修改和实验补强必须配套推进。**

---

## 四、对标 `AI_Based_Sensor`，真正要学的是“实验结构强度”

这里需要明确一个重要判断：

**对标 `AI_Based_Sensor` 时，最应该学习的不是“有多少个模型名”，而是它的实验组织方式。**

它真正强的地方在于：

1. **横评有层次**：不是随便堆几个模型，而是让对比对象覆盖不同方法范式。
2. **消融有因果性**：每个关键设计都被问到“到底有没有必要”。
3. **验证有严格性**：训练/测试 protocol、统计方式、工作流位置都说得清楚。
4. **结果有系统性**：精度、效率、资源、工程意义放在一起看。

因此，当前 MET 稿件如果要向这个强度靠近，确实应当在实验章节引入更像 AI+工程论文的结构：

- 主比较结果；
- 关键机制消融；
- 更严格的数据划分和统计验证；
- 计算/部署系统评估；
- 最后再把结果解释回 sensor/system value。

这意味着：

**更强的横评和消融实验是必要的，而且是应该主动补的，不只是被动回应审稿意见。**

## 五、当前最需要补强的横评设计

如果目标是接近 `AI_Based_Sensor` 的实验成熟度，那么横评至少应形成一个层次清楚的矩阵，而不是只保留现在的 Wiener、GRU、LSTM、RVTDCNN。

### 5.1 第一层：工程/经典基线

这一层回答：你是不是只比深度学习模型强，而没有和本问题最基础的建模方法比较。

建议保留或强化：

- Wiener
- 传统 force-feedback 方案（即便不能同口径重跑，也要统一放进讨论/对照框架）

意义：

- 证明本文不是脱离工程现实的纯 AI benchmark；
- 把“前馈校准 vs 反馈抑制”的系统价值讲清楚；
- 让 MN 读者看到它与 physical sensor path 的关系。

### 5.2 第二层：常规深度学习基线

这一层回答：为什么不是常规时序模型就够了。

当前已有：

- GRU
- LSTM

建议新增：

- **1D-CNN 或 TCN**

原因：

- 它是当前矩阵里最明显缺失的一类；
- 比直接上 Transformer 更自然；
- 更能回答“为什么不是常规卷积时序模型”。

### 5.3 第三层：物理启发或相关任务 AI 基线

这一层回答：你的方案是否真的优于“同样强调物理先验/线性化”的相关方法。

当前已有：

- RVTDCNN

这一层建议保留，并在文中更明确说明它的角色，不只是“来自别的领域的一个模型”，而是：

- 代表了 physics-guided AI compensation 的一类思路。

### 5.4 第四层：更强 AI 基线（可选但建议至少补一个）

如果资源允许，为了更接近 `AI_Based_Sensor` 的强度，建议再补一个较强但不过度偏离任务本身的现代 baseline，例如：

- 一个轻量 Transformer-like 时序模型；或
- 一个近年传感器/补偿场景下常见的强基线。

这里不建议一口气扩成 model zoo，但**至少补到让审稿人不能再轻易说“比较对象太少、代表性不足”** 的程度。

因此，横评方向上更合理的目标不是“只多一个模型”，而是形成下面这类结构：

- 经典工程方法
- 传统块结构建模
- RNN 类
- CNN/TCN 类
- physics-guided 类
- 可选的 modern strong baseline

这就已经明显更接近 `AI_Based_Sensor` 那类论文的实验层次了。

## 六、当前最需要补强的消融设计

如果目标是接近 `AI_Based_Sensor` 的强度，那么当前最缺的不是更多规模变体，而是真正的机制性消融。

### 6.1 必补：结构级消融

这一层回答：为什么必须是当前这套结构，而不是更简单的替代方案。

建议至少补：

1. **线性前端消融**
   - 无前端 / 简化线性前端
   - Wiener/IIR 线性前端
   - 完整方案

2. **非线性映射消融**
   - MLP
   - spline-MLP 或等参数量普通可学习映射
   - KAN

目的：

- 回答 KAN 是否真的有必要；
- 回答 frequency-response-aware 前端是否真的有效；
- 避免“只是参数调优碰巧更好”的质疑。

### 6.2 必补：旧稿关键机制消融

只要你在稿件里还保留类似 PIKAN / FRIRNN / AFMAE 这些表述，下面这些消融就必须做，否则非常危险：

1. **原始 KAN vs 物理约束 KAN**
2. **仅奇对称 / 仅正定 / 二者结合 / 无约束**
3. **有注入 vs 无注入**
4. **MAE vs AFMAE vs MAE+AFMAE**

这是因为：

- 旧稿已经把这些模块写成核心创新；
- 原审稿意见也正是盯着这些点在问；
- 如果不做，MN 审稿时这些问题大概率还会回来。

所以从现实出发，这部分不能回避。

### 6.3 建议补：部署相关消融

若想让“低复杂度、可部署”更有说服力，建议把 LUT/部署部分也做成更正式的消融，而不只是一个加速结果图。

可以补：

- 无 LUT
- 不同 LUT 点数
- 是否插值
- 精度/Flash/latency 的三维 trade-off

这会让部署部分更像一个系统设计选择，而不是附带展示。

## 七、除了横评和消融，还必须同步加强哪些验证

如果只补横评和消融，但不补验证协议，实验部分仍然会显得不够扎实。

### 7.1 更严格的数据划分

当前 `main.tex` 是随机切分子序列：`docs/paper/main.tex:1026`。

这会带来典型质疑：

- 同一频率/振幅条件下的窗口是否同时进入训练和测试；
- 模型是否只是学会了同条件插值；
- 泛化性是否被高估。

因此建议至少补一组更严格划分：

- 留出完整频率点；
- 或留出完整振幅点；
- 或留出整条 sequence；
- 若可能，留出器件个体更好。

### 7.2 多随机种子统计

`AI_Based_Sensor` 这类成熟论文的强度，不只体现在有表格，也体现在结果不是单次最优值。

建议补：

- 3~5 次随机种子重复；
- 均值 ± 标准差；
- 关键指标的误差条或 boxplot。

### 7.3 统一的系统指标表

当前稿件已经有参数量和 STM32 latency，但还不够像一套完整系统评估。

建议统一整理：

- IONS / SDS / NFDS
- 参数量
- MACs 或近似计算量
- STM32 latency
- Flash 占用
- LUT 配置
- 是否需要额外反馈硬件

这样才能把算法效果和工程代价放在同一张表上讨论。

## 八、主题应当怎么改，才能让新增实验真正服务 MN

既然用户判断“实验部分设计尤其是横评和消融是有必要的”，那就更要强调：这些实验必须服务新主题，而不能继续服务旧版方法稿。

建议主题层面至少做到以下转向：

### 8.1 题目与摘要转向 sensor/system 问题

优先突出：

- electrochemical seismometer / electrochemical vibration sensor
- magnitude-dependent drift
- calibration / linearization
- dynamic range extension
- embedded implementation

而不要再让 FRIKAN、PIKAN、FRIRNN 在标题和摘要占据第一主位。

### 8.2 引言改成“工程问题 -> 现有方法缺口 -> 本文方案”

这样做后，新增的横评和消融就不再是“证明我发明的新模块多厉害”，而是：

- 证明这个 calibration framework 为什么合理；
- 证明哪些设计对 sensor problem 真正必要；
- 证明该方案在 accuracy / efficiency / deployability 上达到工程可接受水平。

### 8.3 讨论部分要把实验结果重新解释回 device/system value

新增实验做完之后，讨论的落点应回到：

- 相比力反馈，它的系统复杂度更低；
- 相比普通序列模型，它的部署性更强；
- 相比单纯黑盒补偿，它更接近可解释的 calibration workflow；
- 相比纯算法优化，它更直接服务传感器动态范围扩展。

这一步非常关键，因为否则再完整的实验矩阵，也仍可能被读成“方法论文”。

## 九、最合理的执行策略：按高强度实验版本推进

既然目标已经明确为：

- **主题必须修改**；
- **横评和消融必须显著补强**；

那么接下来的合理路线不应再是“最小修补版”，而应切换成一个更高强度的 MN 转投方案。

### 9.1 推荐的实验补强版本

建议把目标定为：

1. **主主题重写**：改成 sensor/system-centric。
2. **横评矩阵升级**：至少覆盖经典工程方法、块结构建模、RNN、CNN/TCN、physics-guided baseline，并视资源补一个 modern strong baseline。
3. **消融矩阵升级**：至少覆盖前端、KAN、约束、注入、损失函数、LUT。
4. **验证协议升级**：补严格划分、多 seed 统计、统一系统指标表。
5. **讨论逻辑升级**：所有实验结果都回到 calibration / linearization / deployability 主线。

### 9.2 这条路线的代价与收益

代价是明显更高的：

- 训练和评估工作量会上升；
- 论文结构需要显著重写；
- 实验管理和表格组织要更系统。

但收益也很明确：

- 可以正面回应上一稿最核心的实验短板；
- 可以让稿件更接近 `AI_Based_Sensor` 的实验成熟度；
- 可以让主题修改不是“换个说法”，而是有实验支撑的真正升级。

---

## 十、最终判断

本次 R3.4 的最终判断可以压缩成六句话：

1. **如果目标是对齐 `AI_Based_Sensor` 这类 MN 中较成熟的 AI+工程论文，那么当前稿件确实有必要显著补强横评和消融实验。**
2. **这不是可做可不做的锦上添花，而是让稿件实验部分达到更高说服力的必要动作。**
3. **但这些新增实验不能继续服务旧版 FRIKAN 方法叙事，而必须服务新的 sensor/system-centric 主旨。**
4. **因此，主题修改和实验补强必须同步推进。**
5. **最合理的目标不是无限堆模型，而是建立一套层次清楚、因果明确、验证更严格的横评 + 消融 + 系统评估框架。**
6. **只有这样，当前稿件才更有可能既达到更接近 `AI_Based_Sensor` 的实验强度，又保持 MN 所需要的 physical sensor / calibration 论文形态。**

因此，当前最准确的结论是：

**转投 MN 时，主题修改是必须的；而为了让新主题真正成立，对标 `AI_Based_Sensor` 的实验设计思路，尤其是更强的横评和消融实验，也是有必要的。**
