# R24.1 任务报告：LSTM / RNN 的 QEMU vs TF 高 MAE 原因排查

## 任务目标

排查 `LSTM` 与 `RNN` 在板端验证中 `QEMU-MAE / KEIL-MAE` 明显高于 `TCN / 1DCNN` 的原因，确认是否为 C 推理代码生成错误。

## 结论

本轮结论是：**不是权重导出、缩放口径、gate 顺序或 C 代码生成结构性 bug**，而是当前 native C 模板中为了轻量部署而使用的 **`tanh / sigmoid / silu` 近似激活函数** 在 400-step 递归推理中累积了 `1e-3` 量级的 scaled-domain 误差，随后又被 `SCALER_OUTPUT_DATA_RANGE = 53.37688572` 放大，最终表现为 `e-2 ~ e-1` 量级的 `QEMU-MAE / KEIL-MAE`。

也就是说，当前高 MAE 反映的是**近似激活实现的数值误差**，而不是 export/generate 把模型结构、权重、bias、scaler 接错了。

## 直接证据

### 1. 现有 validation comparison 已显示误差主要发生在激活后的递归链路

抽查现有产物：

- `ex_projects/inference/qemu-c-inference/lstm_u16_e1k_puremae_r8/data/validation_comparison.json`
- `ex_projects/inference/qemu-c-inference/rnnu16_e1k_puremae_r15/data/validation_comparison.json`

关键现象：

#### LSTM

- `input_scaled.mae = 5.12e-07`，说明输入打包与 scaler 口径正确；
- `lstm_hidden.mae = 9.37e-04`，说明误差从递归激活开始累计；
- `dense_output.mae = 5.88e-04`；
- `output_scaled.mae = 9.49e-04`；
- 最终 `overall.mae = 5.07e-02`。

这组数满足：

- `5.07e-02 ≈ 9.49e-04 × 53.37688572`

即 scaled-domain 的 `1e-3` 误差被 output scaler 直接放大到了 `5e-2`。

#### RNN

- `input_scaled.mae = 5.12e-07`；
- `dense_output.mae = 5.11e-03`，明显高于 LSTM；
- `output_scaled.mae = 1.64e-03`；
- 最终 `overall.mae = 8.74e-02`。

同样满足：

- `8.74e-02 ≈ 1.64e-03 × 53.37688572`

因此 RNN 更差的直接原因是：`SimpleRNN(tanh) + Dense(silu)` 这条链路对近似激活更敏感，scaled-domain 误差本身就比 LSTM 更大。

### 2. 用“同一份导出权重 + 同一份验证输入”做 exact/approx 对照，能完全复现实验现象

我额外写了一个一次性 Python 分析脚本（未落库），直接复用：

- `projects/01_LR_STUDY/LSTMu16_e1k_puremae_r8/data/best_val.weights.json`
- `projects/07_RNN/RNNu16_e1k_puremae_r15/data/best_val.weights.json`
- 对应 EP `config.json` 的验证输入与 scaler

并分别在 Python 中实现了两套前向：

1. **approx**：严格按当前 C 模板的 `tanh_approx / sigmoid_approx / silu_approx` 前向；
2. **exact**：保持完全相同的结构、权重、bias、scaler，只把激活换成精确 `tanh / sigmoid / silu`。

结果如下：

### LSTM

- approx:
  - `output_scaled_mae = 9.498950676061213e-04`
  - `output_mae = 5.070243775844574e-02`
- exact:
  - `output_scaled_mae = 1.1558004331391203e-08`
  - `output_mae = 6.170041615405353e-07`

### RNN

- approx:
  - `output_scaled_mae = 1.6372567042708397e-03`
  - `output_mae = 8.739165961742401e-02`
- exact:
  - `output_scaled_mae = 4.726294733359282e-08`
  - `output_mae = 2.52270842793223e-06`

这说明：

- **只要保留当前近似激活，Python 就能几乎一比一复现当前 QEMU/KEIL 的高 MAE；**
- **只要把激活改成精确实现，哪怕结构、权重、bias、scaler 完全不变，误差会立刻降到 `1e-6` 量级。**

因此可以排除以下怀疑：

- 权重 JSON 导出错误；
- gate 顺序错误；
- bias 拼接错误；
- Dense / output 层接线错误；
- scaler 前后处理接错；
- validation 输入打包错误；
- QEMU 与 KEIL 之间平台差异。

## 根因定位到的代码位置

当前近似激活实现在：

- `src/core/board_inference/templates/models/lstm_main_template.c`
- `src/core/board_inference/templates/models/rnn_main_template.c`

其中：

- `tanh_approx()`
- `sigmoid_approx()`
- `silu_approx()`

都属于轻量近似实现，不是 TensorFlow 的精确激活。对卷积模型或非递归路径，这类误差通常不会明显积累；但对 400 step 的 `LSTM / RNN` 来说，hidden state 会把每一步的小误差继续带入下一步，因此最终在 `output_scaled` 层面放大到 `1e-3` 量级，最后再乘 `output_data_range` 变成显著的物理域 MAE。

## 对“是否存在 generate bug”的回答

**回答：没有发现 generate/export 结构性 bug。**

当前 `LSTM / RNN` 的高 `QEMU-MAE / KEIL-MAE`，本质上是 native C 版本为了 MCU 轻量实现而采用近似激活函数后带来的数值一致性代价，而不是模型被生成错了。

## 为什么本轮没有直接改代码

理论上可以把 native C 模板的递归激活替换为更精确的实现（例如精确 `tanh/sigmoid/silu`，或更高精度 LUT/近似），这样可以显著降低 `LSTM / RNN` 的 `QEMU-MAE / KEIL-MAE`。

但这会同步改变：

- `QEMU-MAE / KEIL-MAE`；
- `KEIL speed (points/s)`；
- 论文当前横评里与这些模型相关的部署数据。

这已经超出“定位 R24.1 原因”的最小闭环，属于后续是否要调整 native C 数学实现与是否重跑相关板端 benchmark 的决策问题。因此本轮先把原因查实并固定证据，不擅自改动现有 benchmark 口径。

## 建议的后续动作

如果后续希望**真正降低 LSTM / RNN 的 QEMU/KEIL-MAE**，建议作为后续独立任务处理：

1. 为 recurrent native C 模板提供“exact math / fast approx”两种模式；
2. 至少对 `LSTM / RNN / GRU / LSTMTransformer` 这组依赖递归激活的模型统一重跑 QEMU / KEIL；
3. 再决定论文里是保留“当前 fast-approx 口径”，还是改成“更高一致性但可能更慢”的 exact 口径。

R24.1 已完成。本轮完成了：

- 复核现有 LSTM / RNN validation 产物；
