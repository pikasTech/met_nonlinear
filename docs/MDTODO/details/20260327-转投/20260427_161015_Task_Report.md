# R24.5 任务报告

## 任务目标

静态分析 `Table 5 Numerical overview of Wiener-KAN ablation studies.` 中 `System / trainable` 与 `Random / trainable` 两行为什么一旦带 `trainable` 就出现严重劣化，重点核查是否存在“把分块对角 `DIAGIIR` 实现直接当作可训练 RNN 使用”而破坏 MISO RNN 语义的问题。

约束：**只做静态分析，不跑训练/评估代码。**

## 审查范围

- 表格来源：`docs/paper/src/paper_pipeline.py`
- 当前表格产物：`docs/paper/data/generated_tables.md`、`docs/paper/latex/tables/ablation_overview.tex`
- FRIKAN 实现：`src/models/frikan_models.py`
- fast path / 权重保存加载：`src/models/base_models.py`

- 回归测试：`src/tests/models/test_models.py`
- 相关项目：
  - `projects/08_IIR_AB/FRIKANh8u6l6_e1k_lr2e4_sysiir_trainable`
  - `projects/08_IIR_AB/FRIKANh8u6l6_e1k_lr1e4_randiir_trainable`
  - 对照项目：`projects/08_IIR_AB/FRIKANh8u6l6_e1k_lr1e4_randiir_trainable_trueiir_rawfast_mb16`

## 核心结论

当前 Table 5 里的 `System / trainable` 和 `Random / trainable` **不是有效的“trainable-IIR”证据**。根本原因不是论文最近重算指标的方法，而是：

1. **表格仍引用了 2026-04-19 下午修复 trainable-IIR fast path 之前训练出来的旧项目。**
2. **这些旧项目的训练路径并没有把可训练 IIR 正确放进训练图里，而是走了一个错误的 fast path。**
3. **用户怀疑的“分块对角 `DIAGIIR` 训练后会破坏零块结构”在理论上完全成立，但对当前 Table 5 这两个 legacy 项目的直接主因，更准确地说是“旧 fast path 让 trainable-IIR 语义失真”。**

换句话说，**当前表中这两行更像是“修复前错误实现的遗留产物”，而不是“trainable IIR 本身必然炸掉”的可靠实验结论。**

## 证据链

### 1. Table 5 现在确实指向旧项目

`docs/paper/src/paper_pipeline.py:191` 到 `docs/paper/src/paper_pipeline.py:194` 当前写死：

- `System prior, trainable` -> `projects/08_IIR_AB/FRIKANh8u6l6_e1k_lr2e4_sysiir_trainable/data/metrics.json`
- `Random, trainable` -> `projects/08_IIR_AB/FRIKANh8u6l6_e1k_lr1e4_randiir_trainable/data/metrics.json`

对应生成后的表格产物 `docs/paper/data/generated_tables.md:52` 和 `docs/paper/data/generated_tables.md:53` 也是：

- `System prior, trainable`: `94.87 / 64.68 / 7.478`
- `Random, trainable`: `102.57 / 16.65 / 5.144`

### 2. 这两个项目的训练时间早于修复提交

trainable-IIR fast path 修复提交为：

- commit `a7f8bb72`
- 时间：**2026-04-19 15:49:13 +0800**
- 提交说明：`Add ablation results and fix trainable IIR fast path`

而当前表格引用的两个项目，其 `training_state.json` 时间分别是：

- `projects/08_IIR_AB/FRIKANh8u6l6_e1k_lr1e4_randiir_trainable/data/training_state.json`
  - `timestamp = 20260419112251.2`
- `projects/08_IIR_AB/FRIKANh8u6l6_e1k_lr2e4_sysiir_trainable/data/training_state.json`
  - `timestamp = 20260419115710.8`

也就是说，**这两组权重是在 2026-04-19 11:22 / 11:57 训练出来的，明显早于 15:49 的 fast path 修复提交。**

这点很关键：虽然今天重新生成的 `metrics.json` 是新的（`generated_at` 为 2026-04-26），但**指标只是基于旧权重重算**；它不能把旧训练语义自动修正回来。

### 3. 修复前的真正问题：`IIR_TRAINABLE=true` 仍然走了错误 fast path

从 `git show a7f8bb72^:src/models/frikan_models.py` 与 `git show a7f8bb72^:src/models/base_models.py` 可见，修复前的逻辑是：

- `FRIKAN` 总是保留 `self.fast_iir = DIAGIIR(...)`
- `fast_model` 的输入不是原始波形，而是一个占位的 `fast_iir_out`
- `BaseModel.fit()` 在 `use_fast_model=true` 时，总是先执行 `iir_features = self.fast_iir(args[0])`
- 然后把这个**预计算出来的特征张量**喂给 `self.fast_model.fit(...)`

这意味着修复前的 trainable-IIR 路径有两个根本问题：

1. **IIR 层没有留在 `fit()` 的计算图里。**  
   梯度只会更新 `fast_model` 内部的 KAN/MLP 部分，不会沿着 `self.fast_iir(args[0])` 这一步回传。

2. **主模型里的 `self.iir`（`SIMOIIR`）和训练时实际喂特征的 `self.fast_iir`（`DIAGIIR`）不是同一条训练语义路径。**  
   结果就是：配置里写了 `IIR_TRAINABLE=true`，但旧 fast path 并没有形成一个“真正把 IIR 前端一起训练”的实验。

所以，对当前 Table 5 里的这两个旧项目，更准确的描述不是“trainable IIR 训练炸了”，而是：

- **trainable-IIR 当时被错误 fast path 截断了；**
- **表里保留下来的其实是修复前 hybrid 语义下得到的 legacy 权重。**

### 4. 用户怀疑的“分块对角训练破坏 MISO 结构”是成立的，但属于更深一层的结构风险

即便不看旧 fast path，仅从 `src/experimental/mimoiir.py` 的实现也可以看出，用户的担心有充分依据：

- `DIAGIIR` 在 `src/experimental/mimoiir.py:184`、`src/experimental/mimoiir.py:188` 先构造块对角 `A_block_diag` / `B_block_diag`
- 然后在 `src/experimental/mimoiir.py:207` 创建的是**普通全连接 `SimpleRNN`**
- 再在 `src/experimental/mimoiir.py:219` 用 `set_weights([W_xh, W_hh])` 把它初始化成块对角

问题在于：**这里只有初始化，没有结构约束。**

也就是说，如果把 `DIAGIIR(units=8, trainable=true)` 真正放进训练图：

- `kernel` 与 `recurrent_kernel` 仍然是全矩阵可训练参数；
- 对角外原本为 0 的块**不会被 mask 固定**；
- 训练后完全可能长出跨通道耦合项；
- 一旦出现这种耦合，模型就不再等价于“8 路互不耦合、共用同一输入的 MISO IIR 组”。

因此，**“直接训练多路块对角 `DIAGIIR` 会破坏 MISO RNN 正确性”这一判断是对的。**

但要区分两层含义：

- **理论结构风险**：直接训练 `DIAGIIR(units>1)` 会丢掉块对角不变量；
- **当前 Table 5 两行的直接污染源**：它们来自修复前旧 fast path，实验本身就不是语义正确的 trainable-IIR 训练。

### 5. 当前代码已经显式修复到“trainable 时不用块对角快路径”

当前仓库的修复后逻辑非常明确：

- `src/models/frikan_models.py:223`  
  `self.fast_iir = self.iir if iir_trainable else fast_iir`
- `src/models/frikan_models.py:259` 到 `src/models/frikan_models.py:261`  
  `iir_trainable=true` 时，`fast_model` 改为直接吃原始输入，并设置 `fast_model_uses_raw_input = True`
- `src/models/base_models.py:431` 到 `src/models/base_models.py:446`  
  只有固定 IIR 才允许预计算 `iir_features`；trainable-IIR 会直接 `fast_model.fit(*args, **kwargs)`
- `src/tests/models/test_models.py:2082` 起新增回归测试，明确要求：
  - trainable-IIR 的 fast path 仍可保留；
  - 但 `fast_model` 必须吃 raw input；
  - 且 `fast_model` 内不能偷偷保留 `DIAGIIR` 块对角层。

这说明仓库当前维护者已经明确承认并修正了旧问题：**trainable-IIR 不能再沿用“预计算块对角 IIR 特征 -> 训练 fast_model”的老快路径。**

### 6. 一个直接的静态旁证：修复后同 LR 的 random-trainable 项目不再出现这种频漂爆炸

虽然本任务不跑代码，但仓库里已经存在一个修复后对照项目：

- `projects/08_IIR_AB/FRIKANh8u6l6_e1k_lr1e4_randiir_trainable_trueiir_rawfast_mb16/data/metrics.json`

它与 Table 5 当前 `Random / trainable` 行共享相同的关键设定：

- `IIR_TRAINABLE = true`
- `IIR_INIT_BY_SYSTEM = false`
- `learning_rate = 1e-4`

但它是修复后真正的 raw-fast / shared-SIMOIIR 路径，现有结果为：

- `Freq Drift = 4.87 Hz`
- `Sens Drift = 31.90 %`
- `Linearity = 3.528 %`

而 Table 5 当前引用的旧项目：

- `projects/08_IIR_AB/FRIKANh8u6l6_e1k_lr1e4_randiir_trainable/data/metrics.json`

结果是：

- `Freq Drift = 102.57 Hz`
- `Sens Drift = 16.65 %`
- `Linearity = 5.144 %`

这说明至少有一点是确定的：

- **“一旦 trainable 就必然频漂爆炸”并不成立；**
- **当前 Table 5 那条 `Random / trainable` 的异常，主要来自旧实现路径，而不是 trainable 这个标签本身。**

## 根本原因归纳

综合以上证据，R24.5 的静态结论如下：

1. **当前论文 Table 5 的 `System / trainable` 和 `Random / trainable` 两行，引用的是修复前训练出的 legacy 项目。**
2. **修复前的 bug 不是简单的“trainable 会更难训”，而是 fast path 把 trainable-IIR 语义做错了。**
3. **用户提出的“块对角 `DIAGIIR` 一旦直接训练会破坏零块结构”在实现层面是成立的，因此也正是当前代码改成 `SIMOIIR + raw fast path` 的原因。**
4. **因此，这两行不应再被解释为‘正确 trainable-IIR 消融后得到的可信结论’。**

## 对 Table 5 的含义判断

就论文证据质量而言，当前这两行存在以下问题：

- `System / trainable`
  - 旧 fast path 污染；
  - 只训练到 `251 epoch`；
  - `training_alive` 仍是 `true`，不是干净终态；
  - 不能作为 canonical trainable-IIR 结论。

- `Random / trainable`
  - 虽然跑满 `1000 epoch`，但权重同样来自修复前旧路径；
  - 现有数字只能代表“修复前 legacy 实现”的结果；
  - 不能直接写成“正确 trainable random front-end 的正式结论”。

## 建议

1. **不要再把 Table 5 当前这两行当成有效的 trainable-IIR 消融证据。**
2. 如果论文还要保留这两行，应当：
   - 要么改引用到修复后、语义正确、完整训练收敛的新项目；
   - 要么在拿不到新 canonical 结果前，临时从正式表中移除。
3. 若后续还要继续研究“可训练 IIR 前端”，建议再区分两个层级：
   - **层级 A**：保持当前 `SIMOIIR + raw fast path`，先做语义正确实验；
   - **层级 B**：若想保留“可训练但仍是二阶 IIR 系数”的物理语义，应进一步把参数化改成直接训练 `a1/a2/b0/b1/b2` 或加入显式 mask / constraint，而不是继续依赖普通 `SimpleRNN` 的全矩阵自由更新。

## 最终结论

R24.5 的根本原因已经找到：

- **当前 Table 5 的 `System / trainable` / `Random / trainable` 爆炸，核心不是“trainable IIR 天生不行”，而是论文仍在引用 2026-04-19 15:49 修复前训练出的旧项目；这些项目沿用了错误的 trainable-IIR fast path，语义上并不是有效的 trainable-IIR 消融。**
- **用户担心的分块对角结构失真问题在实现上确实存在，因此当前代码已经改为 trainable 时绕开多路 `DIAGIIR` 块对角快路径；但 Table 5 里的这两行还没有同步切换到修复后的 canonical 实验。**
