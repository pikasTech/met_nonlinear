# R5.1 任务报告：MN 改稿所需的决策清单与决策顺序

## 任务目标

在 `R5` 已形成“修改步骤总计划”的基础上，进一步结合当前仓库里的代码、实验项目、preset、LaTeX 稿件和编译状态，整理：

1. 还需要做哪些关键决策；
2. 这些决策应按什么顺序做；
3. 每个决策有哪些候选方案；
4. 基于当前 REPO 状态，推荐选哪个方案，以及为什么。

---

## 一、先给结论：建议按这条顺序做决策

如果只保留一条最务实的决策链，我建议按下面顺序推进：

1. **D1：继续以 MN 为唯一主目标，还是在仓库内开双轨期刊路线**
2. **D2：冻结 claim budget（哪些结论保留，哪些降级）**
3. **D3：冻结正式主实验矩阵（哪些 baseline 进入正文，哪些移到补充）**
4. **D4：决定物理约束这条机制线保留到什么强度**
5. **D5：决定频响先验 / 前端注入这条机制线保留到什么强度**
6. **D6：决定 AFMAE 这一条应当保留为“核心创新”还是“实用设计”**
7. **D7：决定 protocol 要加强到什么程度**
8. **D8：决定 force-feedback 以什么方式进入论文证据链**
9. **D9：决定 `values.md` 和正文表格的唯一数据源**
10. **D10：决定 bibliography / citation pipeline 何时收口**
11. **D11：决定何时开始英文定稿与语言 polishing**

核心原因很简单：

- `D1-D3` 决定论文到底是什么；
- `D4-D8` 决定证据链够不够；
- `D9-D11` 才是把稳定证据写成可提交稿。

也就是说，**先决策论文和证据，再决策写法和润色。**

## 三、按依赖顺序排列的决策清单

下面的顺序，就是我建议的实际决策顺序。

## D1：是否继续以 MN 为唯一主目标

### 候选方案

- **方案 A**：继续以 MN 为唯一主目标，不在 repo 内分叉第二套期刊稿
- **方案 B**：MN 作为主目标，同时在 repo 内并行准备第二套期刊路线
- **方案 C**：暂停 MN，转向更偏方法或测量类期刊

### 推荐

**推荐方案 A。**

### 证据

- 当前仓库已经完成：
  - MN LaTeX 模板接入；
  - `FRIKAN -> Wiener-KAN` 的叙事收敛；
  - 一轮围绕 MN 的 gap 分析与改稿规划。
- 也就是说，仓库已经在 MN 方向上形成明显沉没成本；现在再在 repo 内开双轨，成本高且容易打乱主线。
- 同时，R3/R3.2 的既有结论也不是“MN 不适合”，而是“MN 可以，但要按 sensor/system 论文改写”。

### 这项决策为什么最先做

因为它决定后面所有 claim、baseline、写作结构都按哪套标准来冻结。

---

## D2：claim budget 采用激进版还是保守版

### 候选方案

- **方案 A**：保留强模型创新叙事，继续强调约束、注入、AFMAE 都是核心创新
- **方案 B**：采用保守版 claim budget，只保留当前证据最稳的主张
- **方案 C**：极度保守，只保留数据集和工程现象，不强写模型优势

### 推荐

**推荐方案 B。**

建议保留的主张：

- 前馈补偿 / 校准框架的工程价值；
- Wiener-KAN 作为结构化实现；
- 频漂 / 灵敏度漂移抑制；
- 嵌入式部署潜力。

建议降级的主张：

- 物理约束的完整机制性结论；
- 频响先验“绝对必要”的强结论；
- AFMAE 的普适性或理论性扩张结论；
- 泛化 / 鲁棒性的大范围声称。

### 证据

- 当前 repo 里：
  - loss 线证据较强；
  - constraint 线有部分强证据；
  - frequency-prior 线只有 `randfrirnn`，还没闭环；
  - protocol 线明显偏弱。
- 所以现在最合理的不是“全留”或“全砍”，而是**只保留目前最稳的那部分结论**。

## D3：正式主实验矩阵怎么冻结

### 候选方案

- **方案 A**：沿用当前正文里的旧矩阵（Wiener / GRU / LSTM / RVTDCNN）
- **方案 B**：改用当前 preset 驱动的中等规模矩阵
- **方案 C**：把 repo 里所有能跑的模型尽量都塞进正文

### 推荐

**推荐方案 B。**

建议正文主矩阵保留：

- Wiener-KAN
- Wiener
- LSTM
- GRU
- 1DCNN
- TCN

建议补充材料或附录保留：

- RNN
- LSTMTransformer

建议暂不把 RVTDCNN 放进正文主表，除非先完成“项目来源重新钉死”。

### 证据

- `1DCNN` 和 `TCN` 已经在当前 preset 中，且指标足以进入正式比较，不应再缺席正文。
- `RNN` 和 `LSTMTransformer` 当前数值更像补充对照，不是正文必须核心对象。
- 当前正文写 RVTDCNN，但当前 official preset 里并没有 RVTDCNN，这意味着**正文与正式实验资产的 provenance 已经脱节**。

### 为什么不推荐方案 C

因为 MN 需要的是“层次清楚的比较”，不是 model zoo。

## D4：物理约束这条线保留到什么强度

### 候选方案

- **方案 A**：继续强写完整 constraint effectiveness，并补更多约束组合实验
- **方案 B**：基于 repo 现有证据保留“约束有效”的中等强度结论，但不扩成完整四组合理论故事
- **方案 C**：完全放弃约束相关结论

### 推荐

**推荐方案 B。**

具体执行上建议：

1. 把 `nopositive` 补进结构消融 preset；
2. 保留 `nosym` 和 `nopositive` 两组现成证据；
3. 正文写成“两个约束去掉后都明显退化，因此当前约束设计有工程意义”；
4. 不要写成“我们已经完整证明了全部约束组合的严格最优性”。

### 证据

- `nosym_r5` 已明显退化；
- `nopositive` 退化更严重，而且已经存在于 repo；
- 但 repo 中没有“完全无约束”版本，也没有完整四组合矩阵。

所以最合理的做法不是放弃这条线，而是：

**把现有强证据用起来，但不把结论写得超过证据边界。**

## D5：频响先验 / 前端注入这条线怎么处理

### 候选方案

- **方案 A**：直接保留强主张，按现在正文继续写
- **方案 B**：暂时弱化表述，只保留“正确先验优于随机先验”
- **方案 C**：先补一个“无注入 / 简化前端”项目，再决定最终写法

### 推荐

**推荐方案 B + C 的组合策略：**

- 立即在正文层面按 **方案 B** 控制表述强度；
- 若后续还能补一个关键实验，则优先补 **方案 C**。

### 证据

- 当前 repo 只有 `randfrirnn_r2`，它能证明“随机先验不好”；
- 但 repo 里没有：
  - `*noinject*`
  - `*nofri*`
  - 明确的“无前端 / 简化前端”版本
- 所以当前还不够支撑“前端注入不可替代”的强结论。

换句话说：

**现在可以证明“正确先验比乱来好”，但还不能完整证明“没有先验一定不行”。**

## D6：AFMAE 这条线保留为“核心创新”还是“实用设计”

### 候选方案

- **方案 A**：保留为核心创新之一
- **方案 B**：保留，但定位为“实用且有效的设计选择”
- **方案 C**：弱化到只在方法里简单带过

### 推荐

**推荐方案 B。**

建议写法是：

- 保留“MAE+AFMAE 的组合损失在当前任务上明显优于纯 MAE / 纯 AFMAE”；
- 不要扩写成“AFMAE 本身是普适的新损失范式”。

### 证据

- 当前仓库里，loss 消融的证据是清楚的：
  - 组合损失最好；
  - 纯 MAE 明显更差；
  - 纯 AFMAE 也不如组合。
- 这说明它适合保留，但更适合写成“任务驱动的有效设计”，而不是再拔高成一整条独立创新主线。

## D7：protocol 到底要加强到什么程度

### 候选方案

- **方案 A**：维持当前随机窗口切分，主要靠文字解释
- **方案 B**：基于现有数据补一组更严格划分
- **方案 C**：等多器件 / 多温度数据具备后再重做 protocol

### 推荐

**推荐方案 B。**

优先级建议：

1. 先在现有数据上补一组更严格划分；
2. 同时在正文明确写出：
   - 单器件；
   - 单环境（25°C）；
   - 未覆盖现场温漂；
   - 未覆盖跨器件泛化。

### 证据

- 当前稿件明确写着随机子序列切分；
- 当前实验环境就是单器件、单环境；
- 仓库里也看不到已经准备好的多温度 / 多器件项目资产。

因此，最现实的加强方式不是等新硬件数据，而是：

**先用当前数据做更严格划分，把可做的 protocol strengthening 先做完。**

**不要把它当成“必须新增的训练项目”，而应当把它当成“必须进入讨论框架的工程对照对象”。**

- 当前引用链还未闭合；
- 现在直接英文化，返工概率极高。

所以最合理的方式是：

**把英文润色放到所有“事实层”和“引用层”稳定之后。**

## 四、如果只执行一条推荐路线，具体应该怎么走

基于当前 REPO 状态，我建议最终路线压缩为下面 8 句话：

1. **继续以 MN 为唯一主目标，不在 repo 内开双轨稿件。**
2. **采用保守版 claim budget：保留工程价值、结构化方案、漂移抑制和部署潜力；下调未闭环机制结论。**
3. **把正文主实验矩阵切到：Wiener-KAN + Wiener + LSTM + GRU + 1DCNN + TCN。**
4. **把 RNN / LSTMTransformer 放到补充材料或附录，RVTDCNN 先不放进正文主表，除非先重新钉死项目来源。**
5. **把 `nopositive` 补进结构消融，constraint 这条线可以保留，但不要过度上升为完整理论故事。**
6. **frequency-prior 这条线先弱写；若还能补一个实验，就优先补“无注入 / 简化前端”。**
7. **protocol 至少补一组更严格划分；force-feedback 改为工程对照表，不新开训练项目。**
8. **等以上都冻结后，再统一刷新 `values.md`、修 citation pipeline、最后做英文定稿。**

## 五、最终结论

R5 给出的总原则是“先上游、后下游”。  
R5.1 在当前仓库状态下，把这个原则具体化成一句话就是：

**先做“论文目标 + claim + 正式实验矩阵”的决策，再做“机制补强 + protocol + 工程对照”的决策，最后才做“宏变量、引用、英文稿”的决策。**

对当前这个 repo 来说，真正的上游不是摘要，也不是语句润色，而是：

1. **MN 主目标是否继续保持单轨；**
2. **哪些结论继续保留；**
3. **哪些项目才算正式正文证据；**
4. **哪些机制线需要补，哪些机制线只应降级书写。**

这些问题不先决定，后面所有写作动作都会返工。  
这些问题一旦决定，后面的正文、表格、摘要、结论和英文定稿才会自然收敛。
