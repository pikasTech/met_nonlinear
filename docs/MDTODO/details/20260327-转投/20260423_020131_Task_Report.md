# R10.1 任务报告：调查 TCN / LSTMTransformer 静态复杂度与板端时延趋势不一致的原因

## 任务目标

根据 `docs/MDTODO/20260327-转投.md` 中 R10.1 的要求，本轮任务要对主横评里 `Compute Cost` 与 on-board `KEIL-SPEED` 的趋势关系做进一步调查，重点回答两个问题：

1. 为什么 Wiener-KAN、GRU、LSTM 的静态复杂度与板端时延趋势看起来较一致，而 TCN 与 LSTMTransformer 看起来明显不一致；
2. 这种不一致到底来自静态分析公式本身、模型层级支持缺失，还是来自数据链路中的旧产物或口径混用。

本任务的签收标准不是强行把所有模型做成严格单调，而是：

- 要么定位并修复真正的静态分析问题；
- 要么明确说明“不一致”的根因，并把 draft 里的写法修正到与证据一致。

---

## 调查过程

### 1. 先核对分析模块与默认配置是否一致

本轮首先核查了静态复杂度默认成本模型的两个入口：

- `src/analysis/model_compute_analysis.py`
- `src/config.py`

结果发现：

1. 分析模块默认值早已更新为 `add:multiply:MAP = 1:4:23`；
2. 但 `Config` 层的默认 `compute_cost_model` 仍停留在旧的 `1:1:6`；
3. `cli.py -m` / `--metrics` 等默认流程会把 `Config` 里的默认值继续传给 compute analysis；
4. 这会导致 project 重新导出时，虽然说明文字已经写成 `1:4:23`，实际落盘的 `compute_analysis.json.platform_cost_model` 却仍可能是旧 `1:1:6`。

也就是说，R10.1 里最先看到的“趋势不一致”，其前提数据本身就已经混入了旧口径产物。

### 2. 修复默认值覆写并重刷 draft 相关项目

在 R10.2 中已完成如下修复：

- 把 `src/config.py` 默认 `compute_cost_model` 从 `1:1:6` 改为 `1:4:23`
- 在 `src/tests/test_fail_fast.py` 中加入回归断言
- 对 draft 相关 project 重跑 `cli.py -m`
- 重生成 `generated_tables.md`、`paper_data.json` 与相关图表

修复后再次抽查代表性 project，可以确认当前 `compute_analysis.json.platform_cost_model` 已统一为：

## 调查结果

### 1. TCN 的“明显不一致”已经被消除

在旧稿里，TCN 的 `Compute Cost` 仍停留在旧口径数值 `3904`，但板端时延已经是更新后的正式结果 `1.069 ms/point`，于是看起来像“静态分析明显低估了 TCN”。

修正默认值覆写并重刷后，TCN 当前结果为：

- `Compute Cost = 10656`
- `KEIL-SPEED = 1.0687 ms/point`

此时它重新成为部署子集里：

- **最高静态复杂度**的模型；
- **最慢板端时延**的模型。

因此，TCN 这部分并不存在当前仍待修复的主公式问题；此前的显著异常主要来自旧默认值覆写导致的陈旧产物。

### 2. LSTMTransformer 只剩轻微反转，不再属于“明显异常”

修复后，LSTM 与 LSTMTransformer 的对比为：

- LSTM：`9216`，`0.7011 ms/point`
- LSTMTransformer：`9676`，`0.6842 ms/point`

也就是说：

- LSTMTransformer 的静态复杂度比 LSTM 高约 `4.99%`
- 但板端时延反而快约 `2.41%`

这确实不是严格单调，但它与旧稿里那种“看起来像完全算错了”的异常不同，原因有三点：

1. 反转幅度已经很小，属于近邻模型之间的轻微交叉；
2. 当前 `1:4:23` 平台模型本身仍是代理模型，标定后对数域 `RMSE` 仍为 `0.1168`，最大相对误差仍可到 `20.22%`；
3. `Compute Cost` 只统计 add / multiply / MAP 三类语义操作，没有显式建模所有内存访问、代码布局和实现细节，因此近邻模型之间出现小幅交叉是允许的。

因此，LSTMTransformer 当前并不能再被写成“静态分析明显错误”的案例，更合理的表述是：**它与 LSTM 已回到同一复杂度-速度区间，仍存在小幅代理误差，但不构成需要单独修公式的明显异常。**

### 3. 本轮没有发现需要继续修的 TCN / LSTMTransformer 层级公式错误

综合本轮调查，当前更可靠的结论是：

1. R10.1 观察到的大部分异常，主因是 adopted model 没有真正进入默认 CLI 刷新链；
2. 修正该默认值并重刷项目后，TCN 的异常已经消失；
3. LSTMTransformer 只剩下落在平台代理误差内的轻微反转；
4. 因此，本轮**没有证据支持继续修改 TCN / LSTMTransformer 的静态分析主公式**。

## 对专题中间稿与 draft 的影响

本轮已将调查结论同步到：

- `docs/reference/paper_compute_cost_calibration_method.md`
- `docs/reference/paper_edge_inference_evaluation_method.md`

同步重点包括：

1. adopted model 不仅要更新分析模块默认值，也要同步更新 `Config` 默认 `compute_cost_model`；
2. 当静态复杂度与板端时延出现明显趋势差异时，先检查 `compute_analysis.json.platform_cost_model` 是否真的落盘为 adopted model；
3. draft 中明确写出：旧稿里 TCN / LSTMTransformer 的显著趋势错位，主要来自旧默认值覆写造成的陈旧产物；
4. 对 LSTMTransformer 当前残留的轻微反转，写成“处于平台代理误差内的近邻交叉”，而不是继续渲染为重大异常。

## 最终结论

R10.1 可以完成，结论如下：

1. **TCN 与 LSTMTransformer 旧稿中的显著趋势异常，主因不是当前静态分析公式错误，而是默认 `compute_cost_model` 仍回退到旧 `1:1:6`，导致 draft 使用了混合口径产物。**
2. **修复默认值覆写并批量重刷后，TCN 的静态复杂度与板端时延趋势已经恢复一致。**
3. **LSTMTransformer 相对 LSTM 只剩小幅反转，幅度处于当前 `Compute Cost` 平台代理模型的允许误差内，不构成需要继续单独修公式的明显问题。**

因此，R10.1 的正确收口方式不是继续改 TCN / LSTMTransformer 的层级公式，而是：

- 修复默认值覆写；
- 重刷相关 project；
- 在专题文档与 draft 中明确说明此前异常的真正来源。
