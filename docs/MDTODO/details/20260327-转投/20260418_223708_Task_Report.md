## 三、结果解读

### 3.1 没有出现“把 LR 调一下就恢复正常”的情况

本轮已经覆盖了：

- 原始点：`7e-4`
- 较低稳定区：`3e-4 / 2e-4 / 1.4e-4 / 1e-4`
- 中高 LR 区：`4e-4 / 5e-4`

结果非常一致：

- 成功训练的几个低 LR 版本，统一指标仍然极差；
- `4e-4` 与 `5e-4` 甚至直接在训练期触发 CUDA illegal address；
- 没有任何一个新 LR 能同时把 `Freq Drift / Sens Drift / Linearity` 三项都拉回到可接受范围。

换句话说：

**`nopositive` 的退化不是“只差一轮 LR 微调”的问题，而是结构约束被去掉之后，本身就进入了明显不稳定区。**

### 3.2 各个 LR 的优劣是“分裂”的，而不是单点统治

本轮没有一个 LR 同时在所有指标上最优：

- `7e-4`：`val_loss` 和 `Linearity` 最好；
- `2e-4`：`Freq Drift` 最小；
- `1e-4`：`Sens Drift` 最小；
- `3e-4`：在新增稳定变体里 `val_loss` 最低。

这说明这条消融在去掉 `only_positive` 后，已经处于明显的“高波动、多局部坏点”状态；
不是简单存在一个显然更好的 LR 却还没试到。

### 3.3 中高 LR 区存在稳定性边界

`4e-4` 和 `5e-4` 连续两次在训练早期触发 `CUDA_ERROR_ILLEGAL_ADDRESS`，而 `3e-4` 仍可完整跑完。
在当前环境（RTX 2060, tf26, 当前代码路径）下，可以把 `3e-4` 视作这条 `nopositive` 变体较接近的稳定上界，继续往上没有明显收益，反而更容易直接崩掉。

本轮 `R6.1` 已经完成了足够数量、且方向明确的 LR 补调。结论如下：

## 六、代码实现补查结论

在完成 LR 补调后，又进一步排查了 `nopositive` 的代码实现，重点核查：

- `model_subcfg.only_positive/use_even/use_symmetry` 是否真的传到了 `FRIKAN` 的所有 `DenseKAN` 层；
- `only_positive=false` 是否只是“名字改了”，但前向里仍被强行投影成正值；
- 训练路径与边缘/LUT 近似路径对 `only_positive` 的语义是否一致。

排查结果如下：

1. **没有发现 `nopositive` 配置失效或接线错误。**
   `FRIKAN` 会把 `model_subcfg` 中的 `only_positive/use_even/use_symmetry` 原样传给输出层和所有内层 `DenseKAN`。
2. **`only_positive=false` 在实现上是生效的。**
   当前 `DenseKAN` 的对称分支逻辑是：
   - 先对输入取绝对值；
   - 只有在 `only_positive=true` 时，才把 `spline_kernel` 投影为绝对值；
   - `use_even=false` 时，再乘回输入符号形成奇对称输出。
3. **最小化验证也证明了这个分支确实会分叉。**
   手工给 `DenseKAN` 指定带负号的 spline 系数后：
   - `only_positive=true` 会把系数变成全正；
   - `only_positive=false` 会保留负系数，并产生不同的奇对称输出。
4. **因此，目前没有证据表明 `nopositive` 的极差结果来自实现 bug。**
   更合理的解释是：去掉 positivity 约束之后，模型允许学习到带负系数、非正定的奇对称样条，训练本身就更容易进入不稳定函数形态。

这里有一个语义上很重要的补充：

- 当前 `nopositive` **不是“完全无约束”**；
- 它的真实语义是：**去掉 positivity，仍保留 odd symmetry**。

也就是说，当前 `nopositive` 更准确的定义应是：

**`only_positive=false, use_symmetry=true, use_even=false`，对应“w/o positivity constraint, keep odd symmetry”。**

## 七、当前对论文是否纳入消融的建议

基于本轮 LR 补调和代码补查，当前更稳妥的论文处理建议如下。

### 7.1 可以放，但不是必须放

从实验意义上说，`nopositive` 是有信息量的：

- 它能说明 positivity 约束不是装饰；
- 而且这条结论已经不再是“单个坏点”，而是经过多轮 LR 补调后仍然成立。

所以，**从“能不能作为内部机制证据”这个角度看，它是可以放进消融的。**

### 7.2 但如果放进主表，会存在被追问语义的风险

当前最大的风险不是结果太差，而是定义容易被审稿人追问：

- 它不是 fully unconstrained；
- 它仍保留了奇对称先验；
- 如果主文把它写成“去掉所有约束”或“完全无先验”，会留下明显把柄。

因此，若坚持纳入主文，必须同时满足两点：

1. 名称改成更准确的表述，例如：
   - `w/o positivity constraint`
   - `remove positivity, keep odd symmetry`
2. 在正文或表注中明确说明：
   - `only_positive=false, use_symmetry=true, use_even=false`
   - 并补一句“we additionally retuned the learning rate for this ablation and still observed severe degradation”

### 7.3 对当前稿件，更推荐“不放主表，只保留为内部/回复审稿人证据”

结合当前 MN 转投稿件的主线，我现在更倾向于：

- **不把 `nopositive` 放进主文结构消融主表；**
- 保留 `nosym`、`KAN vs MLP` 等语义更干净、解释成本更低的消融；
- 把 `nopositive` 作为内部证据，或在 rebuttal / 补充说明时再启用。

这样做的好处是：

- 避免审稿人把注意力从“方法是否有效”转移到“这个消融是否 truly unconstrained”；
- 仍然保留一条已经补调过 LR、且代码实现已核查过的备份证据链。

### 7.4 如果最终不放，就要同步降级正文措辞

如果 `nopositive` 不进入正式消融表，那么正文里就不应再强写：

- “positivity constraint has been rigorously isolated and proven essential by ablation”

更稳妥的写法应降级为：

- positivity-aware constrained parameterization is adopted for stability;
- current evidence suggests removing positivity leads to unstable behavior, but this line is not presented as a standalone main-text ablation claim.

   这一点必须在后续写作中说清，否则容易被审稿人追问。
4. **如果后续论文需要一个“已经认真补调过”的 `nopositive` 对照点，建议优先采用 `projects/01_LR_STUDY/FRIKANh8u6l6_e1k_lr2e4_nopositive`。**

   它更适合作为内部论证或 rebuttal 备份证据，而不是当前主线里必须硬塞进去的一格。
6. **如果最终不放，就同步降低正文里关于 positivity 的 claim 强度。**

因此，这轮工作的最终价值可以概括为两点：

- 一方面，它排除了“只是没调 LR”这种低层次质疑；
- 另一方面，它也提醒我们：**`nopositive` 这条线更适合做备份证据，而不是当前版本主文里最值得优先展示的消融。**
