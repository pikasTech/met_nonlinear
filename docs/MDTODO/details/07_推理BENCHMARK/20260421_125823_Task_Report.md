### 2.3 新增独立 debug_cli

新增根脚本：

- `C:\work\met_nonlinear_master\debug_cli.py`

新增实现模块：

- `C:\work\met_nonlinear_master\src\core\board_inference\debug_cli.py`

当前支持的独立命令包括：

```powershell
python debug_cli.py compare <ep_path> --flow qemu --mode generate
python debug_cli.py run-new <ep_path> --flow qemu --mode generate
python debug_cli.py run-legacy <ep_path> --flow qemu --mode generate --old-runner cli
```

设计要点：

- 不接入主 `cli.py`；
- 会把原 EP 目录克隆到 `debug_cli/<timestamp>_<label>/legacy|new/` 隔离目录下执行；
- `compare` 会分别跑 legacy/new 两侧，再对产物目录做归一化比对；
- `old-runner=cli` 时可以直接走旧的 `cli.py ep ...` 主流程，而不是只走模块调用；
- 默认 `--mode generate`，优先验证**稳定生成物**，避免 QEMU / Keil toolchain 依赖影响初期比较。

## 3. 针对 `.wave` 产物的比对修正

### 3.1 初始问题

第一次真实比对时，`debug_cli compare` 在 `lstm_u16_base` 上先暴露了一个假阳性：

- 同样的 legacy 逻辑跑两次，`.wave` 文件字节并不稳定；
- 根因不是波形数据不同，而是压缩包里的 metadata 含有 `creation_date` / `modified_date` 时间戳；
- 如果直接做字节级 diff，会把这种“内容一致、封装时间不同”的情况误判为不一致。

### 3.2 本轮修正

为此，本轮补了 `C:\work\met_nonlinear_master\src\core\board_inference\comparison.py` 的 `.wave` 专门逻辑：

1. 用 `numpy.load(..., allow_pickle=True)` 读取 `.wave` 归档；
2. 对 `record_*` 数据按数组 `shape + dtype + sha256(data)` 做内容比较；
3. 对 `metadata` 先解析成结构化对象；
4. 将 `creation_date` / `modified_date` 归一化为占位符 `__TIMESTAMP__`；
5. 再进行新旧一致性比较。

这样之后，`.wave` 比较口径就变成了“看真实数据和稳定 metadata”，而不是“看 zip 包字节”。
