## 5. 具体拆分映射建议

### 5.1 入口编排层

当前：

- `execute_qemu_inference_task(...)`
- `execute_qemu_inference_keil_bench_task(...)`

建议：

- 在 `entrypoints.py` 中保留这两个公开入口。
- 新增一个共享的 job context 构建步骤，例如：
  - 解析 model project
  - 识别 model type
  - 加载 model spec
  - 准备 validation artifacts
  - 统一生成 output summary 初始结构

这样 QEMU-only 和 QEMU+Keil 两条路径共享同一份准备逻辑，减少当前两段入口之间的复制。

### 5.2 模型注册层

当前：

- `_detect_model_type(...)`
- `_load_model_spec(...)`
- `_load_lstm_model_spec(...)`
- `_load_grn_model_spec(...)`
- `_load_lstm_transformer_model_spec(...)`
- `_load_frikan_model_spec(...)`
- `_load_conv_stack_model_spec(...)`
- `_load_tcn_model_spec(...)`
- `_load_wavenet_model_spec(...)`

建议：

- 新增 `models/registry.py`：
  - 统一做 model type detect
  - 根据类型分发到各模型模块
- 每个模型模块只暴露两类能力：
  - `load_spec(...)`
  - `render_artifacts(...)` 或 `render_main_c(...)` + `render_model_data_h(...)`

这样“模型识别”与“模型实现”解耦，后续新增模型时不需要继续把一个总文件改得更长。

### 5.3 平台层

当前：

- `generate_qemu_project(...)`
- `generate_keil_project(...)`
- `_run_keil_build_job(...)`
- `_run_keil_program_job(...)`
- `_start_keil_serial_capture(...)`
- `_finish_keil_serial_capture(...)`
- `_write_keil_benchmark_uvprojx(...)`

建议：

- `platforms/qemu.py`
  - 负责 runtime template 复制
  - 负责 `main.c`、`model_data.h` 落盘
  - 负责 QEMU build/run 调度
- `platforms/keil.py`
  - 负责最小 Keil 工程生成
  - 负责 `.uvprojx` 生成
  - 负责 Keil CLI build/program
  - 负责串口抓取脚本与结果解析

这样 QEMU 与 Keil 可以共享 model artifact，但不再共享同一坨平台 glue 代码。

### 5.4 模板层

用户提出“模板落到独立 `.c/.h/.xml` 文本文件里”的建议是对的，建议优先执行在以下对象上：

- `benchmark_keil_port.c`
- `benchmark_keil_port.h`
- `main.h`
- `stm32f4xx_it.c`
- `stm32f4xx_it.h`
- `.uvprojx` 基础 XML
- serial capture PowerShell 脚本

这里的原则不是“所有 C 代码都模板化”，而是：

- **固定骨架**放模板文件；
- **模型权重数组、层数、维度等真正动态数据**继续由 Python 渲染；
- **少量条件编译差异**通过占位符或小片段插入解决；
- 避免把 400-900 行整段 C 代码永远硬编码在 Python 三引号里。

### 5.5 输出与报告层

当前 stdout 解析、wave 文件写盘、comparison plot 输出都在一个大文件尾部。

建议：

- `parsing.py` 只负责“把 stdout 变成结构化字典”；
- `outputs.py` 只负责“把结构化结果写成 JSON / wave / PNG”；
- `entrypoints.py` 只负责决定什么时候调用它们；
- 不要让“业务入口函数”再直接包含大段绘图和文件写入细节。

## 7.1 测试总原则

这次重构的测试重点不是“覆盖率数字”，而是守住以下契约：

1. **入口契约不变**：旧函数名还能调用。
2. **生成契约不变**：生成文件名、关键宏、关键 JSON 字段不漂移。
3. **解析契约不变**：串口输出仍能被解析成原来的 key/value 结构。
4. **模型契约不变**：相同权重输入下，各模型生成的 C 结构和 benchmark 结果不变。
5. **平台契约不变**：QEMU 与 Keil 仍然共享公共 benchmark 逻辑，但平台 glue 分离。

## 8. 风险点与规避建议

### 8.1 最大风险：模板迁移后行为悄悄漂移

风险：

- 把大段 C / XML 从 Python 搬到模板文件后，最容易出现占位符漏替换、宏顺序变化、路径相对关系变化。

规避：

- 必须配套 golden 测试。
- 每个模板迁移后都要至少比对一次生成物关键片段。

### 8.2 最大风险：测试 patch 路径大量失效

风险：

- 当前测试大量 patch `core.lstm_qemu_ep_task.*`。
- 迁移后若直接改 import 路径，旧测试会大面积失效。

规避：

- 先保留兼容层。
- 先让兼容层转调新实现，等测试拆分完成后再逐步调整 patch 目标。

### 8.3 最大风险：把“整理结构”做成“顺手改行为”

风险：

- 尤其是 `validation`、`FRIKAN`、`debug stage outputs` 这些地方，一旦边拆边顺手优化，很容易让 benchmark 结果改变。

规避：

- 每阶段只允许做“抽取与搬运”，不允许夹带行为改动。
- 若必须修 bug，应单独提交、单独验收，不和纯重构混在一起。
