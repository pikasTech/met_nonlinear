## 背景

按用户最新要求，本轮的目标是验证 `board_inference` 新架构与 `src/core/lstm_qemu_ep_task.py` 的行为一致性；凡是确认旧架构同样存在的问题，只记录、不修复。

## 已确认的共享故障

### 1. `lstm_u16_base` generate 路径旧权重命名不兼容

- 对照命令：`conda.bat run --no-capture-output -n tf26 python debug_cli.py compare ex_projects/inference/qemu-c-inference/lstm_u16_base --flow qemu --mode generate`
- 对照结果：`legacy_success = false`，`new_success = false`
- 旧/新共同故障：都报 `KeyError: '未找到权重项: lstm_cell/kernel'`
- 触发对象：`projects/00_MAE_VS_AFMAE/LSTMu16_base/data/best_val.weights.json`
- 直接原因：该项目的权重名是 `lstm_1/lstm_cell_1/*`，而旧路径 LSTM generate 逻辑按 `lstm_cell/*` 片段查找。
- 结论：这是旧路径已存在的兼容性缺陷；新架构当前经 legacy adapter 复现了同一失败，不在本轮修复范围内。
- 证据：`C:\work\met_nonlinear_master\ex_projects\inference\qemu-c-inference\lstm_u16_base\debug_cli\20260421_143017_compare\comparison_report.json`

### 2. `frikan_h8u6l6_e1k_lr7e4` keil-bench 串口解析混帧

- 对照命令：`conda.bat run --no-capture-output -n tf26 python debug_cli.py compare ex_projects/inference/qemu-c-inference/frikan_h8u6l6_e1k_lr7e4 --flow keil --mode configured`
- 对照结果：`legacy_success = false`，`new_success = false`
- 旧/新共同故障：都在解析验证串口输出时抛出 `ValueError: could not convert string to float ...`
- 表象：`validation_input_scaled_0=` 等分号分隔长序列与其它字段串接后，被按逗号切分的旧解析器当作单个浮点数，导致转换失败。
- 结论：这是旧路径与新架构共享的 keil-bench 解析故障；本轮只冻结记录，不改旧实现，也不在新架构上单独修补该共享缺陷。
- 证据：`C:\work\met_nonlinear_master\ex_projects\inference\qemu-c-inference\frikan_h8u6l6_e1k_lr7e4\debug_cli\20260421_143115_compare\comparison_report.json`

## 当前处理原则

- 共享于旧路径的问题仅记录，不修复。
- 后续 compare 重点区分三类状态：
  - 新旧都成功且产物一致；
  - 新旧都失败且失败点一致（共享故障）；
  - 只有新旧之一失败，或成功但产物差异超出口径（真实重构偏差）。
