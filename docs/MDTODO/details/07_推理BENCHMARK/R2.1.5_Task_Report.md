# R2.1.5 debug_cli 与旧 CLI 包装行为对齐 + FRIKAN 一致性验证报告

- 日期：2026-04-21
- 目标：先对 `ex_projects/inference/qemu-c-inference/frikan_h8u6l6_e1k_lr7e4` 做静态代码路径对照，再修补新旧包装层的不一致，最后用 `debug_cli` 对旧 `cli.py` 主流程与新隔离入口做一次真实 generate 一致性验证。
- 约束：未修改 `src/core/lstm_qemu_ep_task.py`，未把 `board_inference` 接入主 `cli.py`。

## 1. 静态对照结论

### 1.1 旧实际路径

`cli.py ep ...` 的真实执行链路是：

`cli.py` -> `src/core/external_cli_handler.py:handle_ep_command()` -> `_load_config()` -> `_validate_config()` -> `_execute_qemu_c_inference_task()` -> `src/core/lstm_qemu_ep_task.py:execute_qemu_inference_task()`

这一条链路包含两个重要包装层语义：

1. 先走 `config_validator` 的严格配置校验；
2. 由 `external_cli_handler` 捕获异常并转成 `False` / error log，而不是把底层异常直接抛给调用方。

### 1.2 新隔离路径（修复前）

`debug_cli` 的新侧执行链路是：

`debug_cli.py` -> `src/core/board_inference/debug_cli.py` -> `src/core/board_inference/entrypoints.py` -> `src/core/board_inference/legacy_adapter.py` -> `src/core/lstm_qemu_ep_task.py`

修复前这里有两个与旧 `cli.py` 不一致的点：

1. `board_inference.entrypoints` 直接把 config 传给 legacy adapter，绕过了旧 CLI 的 `_validate_config()`；
2. `board_inference.entrypoints` 没有复现旧 CLI 的异常兜底，legacy 层一旦抛异常，会直接让 `debug_cli` 失败退出；
3. `debug_cli compare` 默认 `old_runner=module`，它比较的是“新包装层 vs 同一个 legacy 模块调用”，而不是“新包装层 vs 旧 `cli.py` 主流程”。

对于 `frikan_h8u6l6_e1k_lr7e4` 这类成功路径，上述问题不一定立刻表现为产物 diff；但对于异常路径（例如此前的 `lstm_u16_base`），它会直接导致新旧包装行为不一致。

## 2. 本轮修改

### 2.1 新入口先走同一套配置校验

在 `src/core/board_inference/config.py` 中新增：

- `validate_task_config()`：直接复用 `src/core/config_validator.py` 里的 `validate_visualization_config_data()`，让新隔离入口和旧 `cli.py ep` 走同一套 `qemu-c-inference` 校验逻辑。

### 2.2 新入口补齐异常兜底

在 `src/core/board_inference/entrypoints.py` 中：

- `build_execution_request()` 改为先校验 config 再构建请求；
- `execute_qemu_inference_task()` / `execute_qemu_inference_keil_bench_task()` 补上与旧 `external_cli_handler` 对齐的 `try/except` 兜底；
- 当 legacy adapter 抛异常时，新入口现在返回 `False` 并打印 error log，而不是把异常直接抛穿给 `debug_cli`。

### 2.3 compare 默认对照旧 cli.py 主流程

在 `src/core/board_inference/debug_cli.py` 中：

- `compare_ep_outputs()` 和 `run_single_ep_path()` 的 `old_runner` 默认值从 `module` 改为 `cli`；
- `compare` / `run-legacy` 子命令的 parser 默认值同步改为 `cli`。

这样 `debug_cli compare` 默认就会拿新隔离入口去对照旧 `cli.py ep ...` 的真实包装行为，而不是只对照底层模块函数。

### 2.4 测试补充

更新了：

- `src/tests/core/board_inference/test_entrypoints.py`
- `src/tests/core/board_inference/test_debug_cli.py`

新增覆盖点：

1. 新入口会先走 config 校验；
2. legacy runner 抛异常时，新入口返回 `False` 而不是直接抛出；
3. `debug_cli compare` 的默认 `old_runner` 为 `cli`。

## 4. 结论

本轮并没有发现 `frikan_h8u6l6_e1k_lr7e4` 的 generate 产物级行为差异；真正存在的不一致主要在**包装层语义**：

1. 新入口之前没有复用旧 CLI 的 config 校验；
2. 新入口之前没有复现旧 CLI 的异常兜底；
3. `debug_cli compare` 之前默认没有对照真实旧 `cli.py` 主流程。

这些点现已对齐；在此基础上，`frikan_h8u6l6_e1k_lr7e4` 已完成“静态对照后再做真实 generate compare”的一致性验证。
