## 1. 静态定位与修复

### 1.1 根因定位

静态比对 `FRIKAN` 与其余 sequence 模型的 C 代码模板后，确认 `FRIKAN` 的生成模板存在一个独有偏差：

- `src/core/board_inference/models/sequence.py` 在 `Keil` 目标下会通过 `#if !defined(BENCHMARK_PLATFORM_KEIL)` 屏蔽中间调试波形的串口转储，只保留 `validation_record_*` 与 benchmark 指标输出。
- `src/core/board_inference/models/frikan.py` 缺少这一层保护，导致 `Keil` 真机运行时仍会把 `input_scaled / iir_output / kan_layer_* / output_scaled` 全量通过串口输出。
- `FRIKAN` 的中间调试矩阵体量远大于其余模型，造成 `validation_record_0` 后续串口流被调试输出污染，最终表现为：
  - 早先 `Keil` 解析失败；
  - 在已有串口解析修复后，`validation_record_0` 仍被后续调试数据拼接污染，导致 `QEMU MAE != KEIL MAE`。

### 1.2 修复内容

本轮修复如下：

1. 在 `src/core/board_inference/models/frikan.py:1077` 为 `FRIKAN` 模板补上与 sequence 模型一致的 `Keil` 侧调试串口屏蔽逻辑，仅在非 `Keil` 平台输出中间调试矩阵。
2. 新增渲染回归测试 `src/tests/core/board_inference/test_frikan_render.py:55`，锁定该保护逻辑，避免后续回归。
