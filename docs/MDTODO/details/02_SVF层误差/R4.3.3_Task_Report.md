# R4.3.3 硬件协同训练配置命名一致性修复报告

## 1. 任务概述

**任务ID**: R4.3.3
**任务名称**: 修复配置命名不一致问题
**任务状态**: Completed
**完成时间**: 2026-02-03

### 1.1 任务背景

R4.3.2 审查报告指出 R4.3.1 设计报告中存在配置命名不一致问题，需要修复以确保设计的完整性和可实施性。

### 1.2 问题描述

在 R4.3.1 设计报告中，`training_mode` 配置使用了不一致的命名：

| 位置 | 使用的命名 |
|------|-----------|
| 3.1 数据流总体架构图 | `trainable_dense_only` |
| 4.1.1 配置格式设计 | `svf_trainable`, `dense_trainable` |
| 4.3.1 Config类新增方法 | `svf_trainable`, `dense_trainable` |
| 6.1 配置验证函数 | `svf_trainable` |

命名存在以下问题：
1. **风格不统一**: `trainable_dense_only` vs `dense_trainable`
2. **语义不一致**: `trainable_dense_only` 暗示"仅Dense可训练"，但实际意图是指定哪些层可训练
3. **可扩展性差**: 新增其他层类型时命名会变得混乱

---

## 2. 修复方案

### 2.1 命名规范定义

根据 R4.3.2 审查建议，采用统一的命名规范：

| 配置项 | 数据类型 | 默认值 | 说明 |
|-------|---------|-------|------|
| `trainable_svf` | bool | false | SVF层是否可训练 |
| `trainable_dense` | bool | true | Dense层是否可训练 |

**命名原则**:
- 使用 `trainable_` 前缀统一样式
- 后接层类型名称 (`svf`, `dense`)
- 语义清晰：直接表达"哪些层可训练"
- 易于扩展：新增层类型时只需添加 `trainable_{layer_type}`

### 2.2 修复后的配置格式

#### 训练配置 (config.json)

```json
{
  "model_subcfg": {
    "hardware_aware_training": {
      "enabled": true,
      "description": "使用实测SVF参数进行协同训练",

      "measured_svf_source": {
        "type": "fitted_params_file",
        "file_path": "data/measured_svf_params.json"
      },

      "training_mode": {
        "trainable_svf": false,
        "trainable_dense": true
      },

      "validation": {
        "compare_with_ideal": true,
        "log_frequency_response": true
      }
    }
  }
}
```
