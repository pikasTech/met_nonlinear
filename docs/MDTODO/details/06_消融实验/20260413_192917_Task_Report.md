## 一、两篇文献各自给出的信息

### 1. `[VPI]AI_Based_Sensor.md` 的核心信息

这篇文献研究的是：

- 输入：`SEM` 截面图像
- 任务：轮廓识别 / 关键尺寸提取
- 主模型：`VLSet-AE`（物理约束 `VAE + level set`）
- 对比对象：`CNN`、`LSTM`、`SVM`、`Random Forest`、`ResNet`、`GoogleNet`、`AttentionNet`

它的结论不是“哪种时序补偿模型更好”，而是：

- 在**噪声较大、低对比度的 SEM 图像**里，单纯的通用图像/序列模型不够稳；
- 如果任务目标本身带有明确物理几何约束，就可以把这些约束直接写进模型结构或 loss；
- 因而 `VLSet-AE` 这类“物理先验 + 专门输入模态”的模型会优于通用 `CNN/LSTM/ResNet`。

对当前仓库横评的直接启发是：

- 这篇文献**不支持**把 `CNN`、`ResNet`、`GoogleNet`、`AttentionNet` 这一类 2D 图像模型直接补进当前横评；
- 因为我们当前的任务是**时域信号补偿 / 频响指标重算**，不是 `SEM` 图像识别；
- 如果硬把这些模型并进来，比较对象会从“同一输入模态下的补偿模型”变成“跨模态任务混比”，结论会失真。

换言之，这篇文献支持的是“任务相关的结构先验很重要”，而不是“当前横评还缺一个图像模型家族”。

### 2. `[VPI]Chen_2025_DE-LOESS_LSTM_Measurement.md` 的核心信息

这篇文献和当前任务更接近，研究的是：

- 输入：低频率、长序列、受温度漂移影响的 MEMS 传感器时序数据
- 预处理：`DE-LOESS`
- 主模型：`LSTM-Transformer`
- 对比对象：`LSTM`、`GRU`、`RNN`、`XGBoost`、`LR`

它给出的信息非常明确：

1. **传统回归基线**（`LR`、`XGBoost`）可以作为低门槛对照，但对长序列、长程依赖、长期漂移建模不足。
2. **时序递归基线**（`RNN`、`GRU`、`LSTM`）是必须有的一条比较链。
3. **混合结构**（`LSTM-Transformer`）的价值在于同时覆盖局部动态与全局依赖。

因此，这篇文献对当前横评最直接的启发是：

- 如果已经有 `GRU / LSTM / LSTM-Transformer`，那**还应该把 `RNN` 也补齐**，这样递归家族对比才完整；
- `LR / XGBoost` 可以作为文献里的“经典方法”参考，但它们更适合单独放在“特征工程 + 传统回归”子表里，而不适合直接与当前端到端时序补偿模型混在同一主表。

## 二、和当前横评名单对照后的判断

当前 6 个候选已经覆盖了：

- `FRIKAN`：带明确结构先验的主线模型
- `TCN`、`1DCNN`：卷积时序模型
- `GRN(GRU)`、`LSTM`：门控循环模型
- `LSTMTransformer`：循环 + 注意力混合模型

从“文献口径是否完整”这个角度看，当前名单**缺的不是卷积模型，也不是 Transformer 模型，而是最基础的 `RNN` 基线**。

原因有三点：

### 1. `RNN` 是 Chen 这篇文献里的标准对照项，但当前名单没有

当前递归族已经有：

- `GRN(GRU)`
- `LSTM`
- `LSTMTransformer`

但没有：

- `RNN`

这样会导致一个问题：我们能回答“门控循环 vs 混合注意力”谁更好，却不能回答“最原始递归单元 vs 门控递归单元”差多少。

而 Chen 文献恰恰把 `RNN -> GRU -> LSTM -> LSTM-Transformer` 视作一条渐进比较链，这条链条在当前横评中还差最后一块拼图。

### 2. 仓库里已经有现成 `RNN` 实现，补充成本低

代码入口已经存在：

- `src/models/base_models.py:568`：`class RNN`

同一文件里也已有：

- `src/models/base_models.py:528`：`class GRN`
- `src/models/base_models.py:607`：`class LSTMTransformer`

说明 `RNN` 不是一个需要新开发的大工程，而是仓库内已有模型族的一员。  
从横评维护成本看，补一个 `RNN` 明显比新引入外部模型家族更划算。

### 3. 仓库里现有旧 `RNN` 项目指标并不差，值得重放进当前统一口径

已有旧项目：

- `projects/RNNu16/data/metrics.json`

当前能读到的旧口径结果是：

- `Freq Drift = 2.5987 Hz`
- `Sens Drift = 13.4410 %`
- `Total Params = 577`

这个结果至少说明：

- `RNN` 不是明显无效的“凑数模型”；
- 它甚至可能在 `Freq Drift / Sens Drift` 上形成一个很有竞争力的低参数基线；
- 只是该项目目前还是旧产物，`Linearity (%)`、`compute_cost` 等字段不完整，不能直接拿来和当前 6 个新口径项目并表。

因此，与其凭印象排除 `RNN`，不如把它补进统一评估口径后再说。

## 三、我对“是否补模型”的结论

### 结论 1：推荐补 `RNN`，而且它是唯一我明确建议立即补进主横评的模型

推荐级别：**高**

推荐原因：

- 与 `Chen_2025` 的对比设计一致；
- 当前横评名单恰好缺这一项；
- 仓库已有实现，落地成本低；
- 仓库旧指标显示它有可能形成“低参数但不差”的基线；
- 补齐后，递归家族比较链会变成：
  - `RNN`
  - `GRU(GRN)`
  - `LSTM`
  - `LSTM-Transformer`

这条链条会比现在更完整，也更容易写成论文/报告中的一段有说服力的结论。

### 结论 2：不推荐把 `[VPI]AI_Based_Sensor.md` 里的图像模型补进当前主横评

不推荐对象包括：

- `CNN`（2D 图像语义）
- `ResNet`
- `GoogleNet`
- `AttentionNet`
- `VLSet-AE`
- `SVM / RF`（用于该文图像/特征识别语境）

不推荐原因：

- 这些模型服务的是 `SEM` 图像轮廓识别任务；
- 当前仓库主横评服务的是**时序补偿 + 频响指标**；
- 输入模态、监督目标、评估指标都不一致；
- 直接混入同一张横评表，会破坏比较语义。

如果以后新开一条“`SEM` 图像侧的特征提取 / 关键尺寸识别”任务线，再引用这篇文献是合理的；但**不是现在这张主横评表该补的对象**。

### 结论 3：`LR / XGBoost` 不建议直接补进当前主横评，但可以作为单独子表候选

推荐级别：**中低**

原因：

- `Chen_2025` 里确实拿它们做了对照；
- 但那篇文章的输入特征是显式构造的 `[smoothed signal, temperature, delta_temperature, ...]`；
- 当前仓库主横评走的是统一训练 / 评估 / `metrics.json` 链路，比较的是端到端补偿模型；
- 若把 `LR / XGBoost` 硬塞进主表，需要先定义新的特征工程和输出接口，否则不是公平对比。

所以更合理的做法是：

- 若后续想体现“传统方法 vs 深度模型”，就单独开一个“经典特征工程基线”子表；
- 不建议和当前 6 个神经网络模型直接混成同一主表。

### 结论 4：若不只看文献、还想增强仓库内部解释性，可把 `FRIMLP` 作为第二优先级补充

这不是两篇文献直接要求的，但从当前仓库语义上看，它有额外价值：

- `FRIKAN` 已经在主表中；
- `FRIMLP` 是同一 `FRI` 前端下“把 `KAN` 主体替换成 `MLP`”的真消融；
- 它适合回答“优势来自 `FRI` 前端，还是来自 `KAN` 主体”。

不过它和本次“依据两篇文献补横评”的关系没有 `RNN` 那么直接，所以我把它放在第二优先级，而不是第一优先级。

同时，现有 canonical `FRIMLP` 项目：

- `projects/04_FRIMLP/FRIMLPh8u6l6_e1k_lr7e4_mlp20l6_tanh_d00`

其当前统一口径指标为：

- `Freq Drift = 5.8179 Hz`
- `Sens Drift = 27.4803 %`
- `Linearity = 1.0120 %`
- `Total Params = 2493`
- `Compute Cost = 5152`

它更适合作为“结构语义对照项”，而不是“最强性能候选”。

## 四、最终建议排序

如果只补一个模型：

1. **补 `RNN`**

如果补两个模型：

1. **补 `RNN`**
2. **补 `FRIMLP`**（作为仓库内部可解释性对照，而非文献强制项）

如果坚持只按这两篇文献来补，而不引入额外内部消融：

1. **补 `RNN`**
2. **其余先不补**

---

## 五、执行建议

若下一步真的要把它补进横评，我建议按下面顺序做：

1. 先把 `RNN` 复制成新的 project 变体并走完整 `-t / -e / --metrics` 统一链路；
2. 不要直接复用旧 `projects/RNNu16` 的 partial `metrics.json` 当最终横评结果；
3. 如果要补第二个项目，再决定是否把 `FRIMLP` 放进主表；
4. `LR / XGBoost` 若要做，只开单独子表，不和主表混排；
5. 继续不要把图像域模型并入这张时序横评表。

---

## 最终结论

基于这两篇文献和当前仓库已有横评名单，我的结论是：

- **推荐补充：`RNN`**
- **可选补充：`FRIMLP`（仓库内部解释性补充，不是文献刚需）**
- **不推荐补充到当前主横评：`CNN/ResNet/GoogleNet/AttentionNet/VLSet-AE`、`LR/XGBoost`**

因此，R4 的直接输出可以浓缩为一句话：

> 当前主横评最值得补的不是新的卷积或图像模型，而是把递归家族缺失的 `RNN` 基线补齐；其余模型要么任务模态不匹配，要么更适合作为独立子表。
